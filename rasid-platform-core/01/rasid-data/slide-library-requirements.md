# وثيقة متطلبات مكتبة عناصر العروض التقديمية

## منصة راصد البيانات — نظام إدارة مكتبة العروض

**الإصدار:** 2.0  
**التاريخ:** 15 مارس 2026  
**المؤلف:** Manus AI  

---

## 1. نظرة عامة

مكتبة عناصر العروض التقديمية هي المكوّن المركزي في منصة راصد البيانات المسؤول عن إدارة وتنظيم وتوليد عناصر التصميم المستخدمة في إنشاء العروض التقديمية الاحترافية. تعمل المكتبة كمستودع ذكي يربط بين ثلاثة محاور: **تخزين القوالب** في قاعدة البيانات، **المعاينة البصرية** عبر محرك عرض HTML، و**التوليد الآلي** عبر الذكاء الاصطناعي. يتيح النظام للمسؤولين إنشاء عناصر تصميم جديدة يدوياً أو عبر الذكاء الاصطناعي، وتحريرها بصرياً دون الحاجة لكتابة كود، ثم استخدامها تلقائياً عند إنشاء عروض تقديمية جديدة.

تتكامل المكتبة مع محرك العروض التقديمية بالذكاء الاصطناعي (AIPresentationCreator) بحيث يتم جلب القوالب المناسبة تلقائياً بناءً على نوع المحتوى المطلوب، ثم تخصيصها ببيانات حقيقية يولّدها الذكاء الاصطناعي. هذا يضمن أن كل عرض تقديمي يُنتج يتبع معايير التصميم المعتمدة من المكتب الوطني لإدارة البيانات (NDMO) مع الحفاظ على مرونة التخصيص.

## 2. البنية التقنية

### 2.1 هيكل قاعدة البيانات

يعتمد النظام على أربعة جداول مترابطة في قاعدة بيانات MySQL/TiDB، مبنية باستخدام Drizzle ORM:

| الجدول | الوصف | الحقول الرئيسية |
|--------|-------|-----------------|
| `slide_templates` | القوالب المرفوعة (ملفات PPTX الأصلية) | `id`, `name`, `description`, `fileUrl`, `fileKey`, `slideCount`, `elementCount`, `status`, `uploadedBy` |
| `element_categories` | فئات تصنيف العناصر | `id`, `slug`, `nameAr`, `nameEn`, `description`, `icon`, `sortOrder` |
| `slide_elements` | العناصر الفردية القابلة لإعادة الاستخدام | `id`, `templateId`, `categoryId`, `name`, `description`, `designTemplate`, `styleProperties`, `contentSlots`, `htmlTemplate`, `isActive`, `usageCount`, `qualityRating` |

**الحقل الأهم** في جدول `slide_elements` هو `htmlTemplate` الذي يحتوي على كود HTML كامل ومستقل (1280×720 بكسل) يمثل الشريحة بصرياً. هذا الكود يتضمن CSS مدمج، خطوط عربية (Tajawal) من Google Fonts، رسوم بيانية عبر Chart.js CDN، وبيانات نموذجية حقيقية. يُعرض هذا الكود عبر `iframe` مع خاصية `srcdoc` لتقديم معاينة بصرية دقيقة.

## 3. الوظائف والمميزات التفصيلية

### 3.1 لوحة إدارة المكتبة (SlideLibraryAdmin)

لوحة الإدارة هي الواجهة الرئيسية التي يتعامل معها المسؤول لإدارة عناصر المكتبة. تتكون من ثلاثة أقسام رئيسية:

**شريط الإحصائيات العلوي:** يعرض أربع بطاقات إحصائية تُجلب من endpoint `slideLibrary.getStats`:
- عدد القوالب المرفوعة (PPTX files)
- عدد العناصر الإجمالي
- عدد الفئات المتاحة
- عدد العناصر النشطة

**شريط التصفية والبحث:** يتيح للمسؤول تصفية العناصر حسب الفئة عبر أزرار أفقية قابلة للتمرير، مع خيار "الكل" لعرض جميع العناصر. يتم جلب الفئات من endpoint `slideLibrary.getCategories`.

**شبكة بطاقات العناصر:** تعرض العناصر في شبكة متجاوبة (3 أعمدة على الشاشات الكبيرة، عمودان على المتوسطة، عمود واحد على الصغيرة). كل بطاقة تتضمن:
- معاينة بصرية حية عبر `ElementPreview` (iframe مع srcdoc)
- اسم العنصر والفئة التابع لها
- تقييم الجودة (1-5 نجوم)
- حالة التفعيل (نشط/معطل)
- عدد مرات الاستخدام
- أزرار: تحرير القالب، تعديل البيانات، حذف

**آلية العمل:** عند تحميل الصفحة، يتم استدعاء `slideLibrary.getAllElements` الذي يجلب جميع العناصر مع بيانات الفئات المرتبطة. البيانات تشمل `htmlTemplate` الذي يُمرر إلى `ElementPreview` للعرض البصري. عند التصفية حسب الفئة، يتم ترشيح النتائج محلياً دون استدعاء إضافي للخادم.

### 3.2 محرر القوالب المرئي (TemplateEditor)

محرر القوالب هو أداة تحرير بصرية مدمجة تتيح للمسؤول تعديل قوالب HTML دون الحاجة لكتابة كود. يُفتح كنافذة منبثقة ملء الشاشة عند الضغط على زر "تحرير القالب" في بطاقة العنصر.

**التخطيط:** ينقسم المحرر إلى قسمين رئيسيين:
- **الجانب الأيمن (60%):** معاينة حية للقالب عبر iframe بحجم 1280×720 مع تحجيم تلقائي يتناسب مع حجم النافذة
- **الجانب الأيسر (40%):** لوحة أدوات التحرير مع ثلاث تبويبات

**تبويبات التحرير:**

1. **تبويب الألوان:** يستخرج تلقائياً جميع الألوان الفريدة المستخدمة في كود HTML (عبر regex يبحث عن أنماط `#RRGGBB`). يعرض كل لون كزر دائري مع حقل إدخال. عند تغيير أي لون، يتم استبداله في جميع مواضعه في كود HTML فوراً، وتتحدث المعاينة الحية تلقائياً. يتوفر أيضاً شريط ألوان مسبقة الإعداد (COLOR_PRESETS) يتضمن 12 لوناً شائعاً في تصاميم NDMO.

2. **تبويب النصوص:** يستخرج جميع النصوص المرئية من كود HTML (عبر regex يبحث عن محتوى بين `>` و `<` بطول 3 أحرف أو أكثر، مع استبعاد كود JavaScript). يعرض كل نص في حقل إدخال قابل للتعديل. عند تغيير أي نص، يتم استبداله في HTML مع الحفاظ على المسافات البيضاء المحيطة.

3. **تبويب الكود:** يعرض كود HTML الكامل في محرر نصي (textarea) مع خط أحادي المسافة واتجاه LTR. يتيح التعديل المباشر على الكود مع تحديث فوري للمعاينة.

**نظام التراجع/الإعادة (Undo/Redo):** يحتفظ المحرر بمكدس تراجع (undoStack) ومكدس إعادة (redoStack). كل تعديل يدفع الحالة الحالية إلى مكدس التراجع ويمسح مكدس الإعادة. أزرار التراجع والإعادة متاحة في شريط الأدوات العلوي.

**الحفظ:** عند الضغط على زر "حفظ"، يتم استدعاء endpoint `slideLibrary.updateElement` مع `htmlTemplate` المحدّث. يتم إرسال الكود الكامل إلى الخادم الذي يحدّث الحقل في قاعدة البيانات. بعد الحفظ الناجح، يتم إعادة تحميل قائمة العناصر عبر `invalidate`.

**آلية استخراج الألوان:**
```
الخوارزمية:
1. تطبيق regex /#[0-9a-fA-F]{6}/g على كود HTML
2. تجميع النتائج في Set لإزالة التكرارات
3. تحويل Set إلى Array
4. عرض كل لون كزر دائري مع حقل إدخال
```

**آلية استبدال الألوان:**
```
الخوارزمية:
1. تهريب اللون القديم لاستخدامه في regex
2. إنشاء regex عالمي غير حساس لحالة الأحرف
3. استبدال جميع التكرارات في HTML
4. تحديث المعاينة الحية فوراً
```

### 3.3 واجهة إنشاء عنصر جديد (CreateTemplateDialog)

نافذة إنشاء عنصر جديد تتيح ثلاثة أوضاع للإنشاء:

**الوضع الأول — التوليد بالذكاء الاصطناعي:**
- يصف المسؤول القالب المطلوب بالنص الحر (مثال: "شريحة تحتوي على 4 بطاقات KPI بألوان NDMO مع رسم بياني دائري")
- يتم إرسال الوصف إلى endpoint `slideLibrary.convertPptxToHtml`
- الخادم يرسل الوصف إلى نموذج LLM مع تعليمات تصميم محددة (أبعاد 1280×720، خط Tajawal، ألوان NDMO)
- يُعاد كود HTML كامل يُعرض في معاينة حية
- المسؤول يراجع النتيجة ويضغط "حفظ" لإضافتها للمكتبة

**الوضع الثاني — رفع HTML:**
- رفع ملف HTML مباشرة (.html أو .htm) عبر مستعرض الملفات
- أو لصق كود HTML يدوياً في حقل نصي
- يُعرض الكود في معاينة حية قبل الحفظ
- يجب أن يكون الكود بأبعاد 1280×720 بكسل

**الوضع الثالث — قالب فارغ:**
- يُنشئ قالب HTML فارغ بتصميم أساسي (خلفية متدرجة كحلية، عنوان ذهبي، نص فرعي)
- يمكن تعديله لاحقاً عبر المحرر المرئي

**الحقول المشتركة لجميع الأوضاع:**
- اسم العنصر (إلزامي)
- الفئة (اختياري — قائمة منسدلة من الفئات المتاحة)
- الوصف (اختياري)

**آلية الحفظ:** يتم استدعاء endpoint `slideLibrary.createElement` الذي ينشئ سجلاً جديداً في جدول `slide_elements` مع `htmlTemplate` المحدد.

### 3.4 رفع ومعالجة ملفات PowerPoint (PPTX)

يدعم النظام رفع ملفات PPTX وتفكيكها تلقائياً إلى عناصر فردية قابلة لإعادة الاستخدام:

**مراحل المعالجة:**

1. **الرفع إلى S3:** يتم تحويل الملف من Base64 إلى Buffer ورفعه إلى S3 تحت مسار `library/templates/{timestamp}-{filename}`.

2. **تحليل XML:** يتم فك ضغط ملف PPTX (وهو ملف ZIP) باستخدام مكتبة JSZip، ثم قراءة ملفات XML الخاصة بكل شريحة (`ppt/slides/slide{n}.xml`). يتم استخراج:
   - النصوص (عبر regex على عناصر `<a:t>`)
   - وجود رسوم بيانية (`c:chart`)
   - وجود جداول (`<a:tbl>`)
   - وجود صور (`<p:pic>`)
   - عدد الأشكال والكتل النصية

3. **التفكيك بالذكاء الاصطناعي:** لكل شريحة، يتم إرسال بياناتها المستخرجة إلى نموذج LLM مع تعليمات لتفكيكها إلى عناصر تصميم فردية. النموذج يُعيد JSON منظم يتضمن لكل عنصر:
   - `categorySlug`: الفئة المناسبة
   - `elementName`: اسم وصفي
   - `description`: وصف تفصيلي للشكل البصري
   - `designTemplate`: قالب التصميم (تخطيط، ألوان، خطوط، مسافات، بيانات نموذجية)
   - `contentSlots`: فتحات المحتوى المتوقعة
   - `usageContexts`: سياقات الاستخدام

4. **الحفظ في قاعدة البيانات:** يتم إنشاء سجل لكل عنصر في `slide_elements` مع قواعد استخدام في `element_usage_rules`.

5. **تحديث حالة القالب:** يتم تحديث سجل القالب في `slide_templates` بعدد الشرائح والعناصر المستخرجة وتغيير الحالة إلى "ready".

**معالجة الأخطاء:** في حالة فشل التفكيك بالذكاء الاصطناعي لشريحة معينة، يتم إنشاء عنصر واحد افتراضي يمثل الشريحة كاملة مع تصنيف تلقائي بناءً على محتواها (جدول → data_table، رسم بياني → horizontal_bars، غير ذلك → infographic).

### 3.5 المعاينة البصرية (ElementPreview)

مكوّن المعاينة يعمل بنظام أولويات:

**الأولوية الأولى — قالب HTML (htmlTemplate):** إذا كان العنصر يحتوي على حقل `htmlTemplate` غير فارغ، يتم عرضه عبر iframe مع خاصية `srcdoc`. يتم حساب نسبة التحجيم ديناميكياً باستخدام `ResizeObserver` لضمان تناسب الشريحة (1280×720) مع حجم الحاوية. الـ iframe يستخدم `sandbox="allow-scripts"` للسماح بتشغيل Chart.js مع منع الوصول للصفحة الأم.

**الأولوية الثانية — معاينة SVG/HTML مبسطة:** إذا لم يتوفر قالب HTML، يتم عرض معاينة مبسطة بناءً على `elementType` و`designTemplate`. يدعم المكوّن 7 أنواع معاينة:
- **جدول (table/data_table):** جدول مصغر مع رؤوس ملونة وصفوف متناوبة
- **رسم بياني (chart/horizontal_bars/circular_gauge):** أشرطة أفقية مع تسميات ونسب
- **بطاقة KPI (kpi_card/kpi):** شبكة بطاقات مع أيقونات وقيم
- **إنفوجرافيك (infographic):** دوائر مع أيقونات وتسميات
- **تدفق عمليات (process_flow):** مربعات متسلسلة مع أسهم
- **غلاف (cover):** خلفية متدرجة مع عنوان مركزي
- **افتراضي:** خطوط رمادية تمثل محتوى نصي

### 3.6 محرك توليد العروض بالذكاء الاصطناعي (AI Presentation Engine)

محرك التوليد يدعم وضعين للإخراج:

**الوضع الأول — Canvas (الافتراضي السابق):** يولّد شرائح كبيانات JSON يتم عرضها عبر مكونات React مخصصة (LargeSlideCard). يستخدم endpoint `ai.generatePresentation`.

**الوضع الثاني — HTML Templates (الافتراضي الحالي):** يولّد شرائح HTML كاملة باستخدام قوالب المكتبة الحقيقية. يستخدم endpoint `ai.generateHtmlPresentation`.

**آلية عمل وضع HTML Templates:**

1. **تحديد تسلسل التخطيطات:** بناءً على عدد الشرائح المطلوب، يتم تحديد تسلسل التخطيطات الإلزامية:
   - 3-4 شرائح: `title` + `chart` + `closing`
   - 5-7 شرائح: يُضاف `table` + `infographic`
   - 8-11 شريحة: يُضاف `toc` + `kpi` + `timeline`
   - 12+ شريحة: يُضاف `executive-summary` + `pillars`
   - الشرائح المتبقية تُملأ بتخطيطات متنوعة (`content`, `two-column`, `pillars`, `kpi`, `timeline`, `quote`)

2. **جلب القوالب المطابقة:** تستدعي الدالة `getHtmlTemplatesForLayouts` قاعدة البيانات لجلب أفضل قالب HTML لكل نوع تخطيط. يتم ذلك عبر:
   - خريطة ربط بين أنواع التخطيطات وفئات العناصر (`layoutToCategoryMap`)
   - ترتيب العناصر حسب تقييم الجودة (`qualityRating`) تنازلياً
   - اختيار أول عنصر مطابق يحتوي على `htmlTemplate`

3. **توليد المحتوى بالذكاء الاصطناعي:** يتم إرسال الموضوع وتسلسل التخطيطات إلى LLM مع طلب JSON منظم يتضمن لكل شريحة: عنوان، عنوان فرعي، محتوى (60+ كلمة)، نقاط رئيسية (5-7)، ملاحظات، بيانات رسوم بيانية، بيانات جداول، عناصر إنفوجرافيك، وعناصر جدول زمني.

4. **تخصيص القوالب:** لكل شريحة، إذا وُجد قالب HTML مطابق، تُستدعى الدالة `customizeHtmlTemplate` التي تقوم بـ:
   - استبدال العناوين: regex يبحث عن نصوص تبدأ بـ "عنوان" أو "العنوان" ويستبدلها بالعنوان المولّد
   - استبدال العناوين الفرعية: regex يبحث عن "العنوان الفرعي" أو "وصف"
   - استبدال المحتوى: regex يبحث عن "فقرة تحليلية" أو "المحتوى"
   - تحديث بيانات Chart.js: استبدال مصفوفات `data`, `labels`, `backgroundColor` في كود JavaScript
   - تحديث الجداول: استبدال `<thead>` و`<tbody>` ببيانات جديدة
   - استبدال عناصر الإنفوجرافيك (best-effort)

5. **الإرجاع:** كل شريحة تُعاد مع حقل `htmlTemplate` (الكود المخصص) وحقل `hasTemplate` (boolean يشير لوجود قالب).

## 5. التأثيرات والتكاملات

### 5.1 التأثير على محرك العروض التقديمية

عند إنشاء عرض تقديمي جديد عبر AIPresentationCreator، يتأثر سلوك المحرك بمحتوى المكتبة كالتالي:

- **وجود قوالب HTML في المكتبة:** يتم استخدامها مباشرة وتخصيصها بالمحتوى المولّد. الشرائح الناتجة تكون بجودة عالية ومتسقة مع هوية NDMO.
- **عدم وجود قوالب مطابقة:** يعود المحرك لوضع Canvas حيث يتم بناء الشرائح من مكونات React مع بيانات JSON. الجودة البصرية أقل لكن المحتوى يبقى كاملاً.
- **مزيج:** بعض الشرائح تستخدم قوالب HTML والبعض الآخر يستخدم Canvas، مع مؤشر `hasTemplate` لكل شريحة.

### 5.2 التأثير على تقييم الجودة

حقل `qualityRating` (1-5) يؤثر مباشرة على أولوية اختيار القوالب عند التوليد. العناصر ذات التقييم الأعلى تُفضّل عند وجود عدة عناصر مطابقة لنفس الفئة. هذا يعني أن تحسين تقييم عنصر معين يجعله الخيار الافتراضي لجميع العروض المستقبلية.

### 5.3 التأثير على قواعد الاستخدام

قواعد الاستخدام (`elementUsageRules`) تحدد متى يُستخدم كل عنصر عبر حقل `triggerContext`. هذا يتيح ربط عنصر واحد بعدة سياقات (مثلاً: عنصر "بطاقة KPI" يمكن استخدامه في سياق `kpi_display` وسياق `summary`). حقل `priority` يحدد الأولوية عند تعارض عدة عناصر.

### 5.4 التأثير على عدد الاستخدامات

حقل `usageCount` يتتبع عدد مرات استخدام كل عنصر في العروض المولّدة. هذا يوفر للمسؤول رؤية واضحة حول العناصر الأكثر استخداماً والعناصر التي لم تُستخدم بعد، مما يساعد في تحسين المكتبة.

## 6. تدفقات العمل الرئيسية

### 6.1 تدفق إضافة عنصر جديد عبر AI

```
المسؤول → يفتح نافذة "إنشاء عنصر جديد"
       → يختار وضع "التوليد بالذكاء الاصطناعي"
       → يكتب وصفاً تفصيلياً للقالب المطلوب
       → يحدد الاسم والفئة
       → يضغط "توليد بالذكاء الاصطناعي"
       → [الخادم] يرسل الوصف إلى LLM مع تعليمات تصميم
       → [LLM] يُعيد كود HTML كامل (1280×720)
       → المعاينة الحية تعرض النتيجة
       → المسؤول يراجع ويضغط "حفظ"
       → [الخادم] يحفظ العنصر في slide_elements
       → العنصر يظهر في المكتبة ويصبح متاحاً للتوليد
```

### 6.2 تدفق تحرير قالب موجود

```
المسؤول → يضغط "تحرير القالب" على بطاقة العنصر
       → يُفتح المحرر المرئي (TemplateEditor)
       → يعدّل الألوان عبر تبويب الألوان (استبدال فوري)
       → يعدّل النصوص عبر تبويب النصوص
       → أو يعدّل الكود مباشرة عبر تبويب الكود
       → المعاينة الحية تتحدث فوراً مع كل تعديل
       → يمكنه التراجع/الإعادة عبر أزرار Undo/Redo
       → يضغط "حفظ"
       → [الخادم] يحدّث htmlTemplate في قاعدة البيانات
       → التغييرات تنعكس على جميع العروض المستقبلية
```

### 6.3 تدفق رفع ملف PPTX

```
المسؤول → يضغط "رفع قالب PPTX"
       → يختار ملف PPTX من جهازه
       → يحدد اسماً ووصفاً
       → يضغط "رفع"
       → [الخادم] يرفع الملف إلى S3
       → [الخادم] يفك ضغط PPTX ويقرأ XML لكل شريحة
       → [الخادم] يستخرج النصوص ويكتشف أنواع المحتوى
       → [الخادم] يرسل كل شريحة إلى LLM للتفكيك
       → [LLM] يُعيد عناصر فردية مع بيانات تصميم مفصلة
       → [الخادم] يحفظ كل عنصر في slide_elements مع قواعد استخدام
       → [الخادم] يحدّث حالة القالب إلى "ready"
       → العناصر المستخرجة تظهر في المكتبة
```

### 6.4 تدفق إنشاء عرض تقديمي بقوالب HTML

```
المستخدم → يفتح "إنشاء عرض تقديمي"
        → يختار وضع الإخراج "قوالب HTML"
        → يكتب الموضوع ويحدد عدد الشرائح
        → يضغط "إنشاء"
        → [الخادم] يحدد تسلسل التخطيطات الإلزامية
        → [الخادم] يجلب أفضل قالب HTML لكل تخطيط من المكتبة
        → [الخادم] يرسل الموضوع والتخطيطات إلى LLM
        → [LLM] يولّد محتوى غني لكل شريحة (JSON منظم)
        → [الخادم] يخصص كل قالب HTML بالمحتوى المولّد
        → [الخادم] يُعيد مصفوفة شرائح مع htmlTemplate لكل واحدة
        → [العميل] يعرض كل شريحة عبر iframe (HtmlSlideCard)
        → المستخدم يراجع ويمكنه تعديل أو تصدير العرض
```

## 7. المواصفات التقنية للقوالب

### 7.1 مواصفات قالب HTML

كل قالب HTML يجب أن يلتزم بالمواصفات التالية:

| المعيار | القيمة |
|---------|--------|
| الأبعاد | 1280×720 بكسل (نسبة 16:9) |
| الاتجاه | RTL (`dir="rtl"` و `lang="ar"`) |
| الخط الأساسي | Tajawal من Google Fonts (أوزان: 400, 500, 700, 800) |
| الألوان الأساسية | كحلي داكن `#0f2744`، ذهبي `#d4af37`، أخضر مائي `#0CAB8F` |
| الرسوم البيانية | Chart.js من CDN (إذا لزم الأمر) |
| CSS | مدمج بالكامل (inline styles أو `<style>` داخلي) |
| JavaScript | مسموح (لـ Chart.js فقط) |
| صور خارجية | غير مسموح (يجب أن يكون القالب مستقلاً بالكامل) |
| Overflow | `overflow: hidden` على `<body>` |

### 7.2 بنية القالب النموذجية

```html
<!DOCTYPE html>
<html dir="rtl" lang="ar">
<head>
  <meta charset="UTF-8">
  <link href="https://fonts.googleapis.com/css2?family=Tajawal:wght@400;500;700;800&display=swap" rel="stylesheet">
  <script src="https://cdn.jsdelivr.net/npm/chart.js"></script> <!-- إذا لزم -->
  <style>
    * { margin: 0; padding: 0; box-sizing: border-box; }
    body {
      width: 1280px; height: 720px; overflow: hidden;
      font-family: 'Tajawal', sans-serif; direction: rtl;
      background: linear-gradient(135deg, #0f2744 0%, #1a3a5c 100%);
    }
    /* ... باقي الأنماط ... */
  </style>
</head>
<body>
  <!-- محتوى الشريحة -->
</body>
</html>
```

## 9. الأداء والتحسينات

**التحجيم التلقائي:** يستخدم `ResizeObserver` لحساب نسبة التحجيم ديناميكياً عند تغيير حجم النافذة، مما يضمن عرض القوالب بشكل صحيح على جميع أحجام الشاشات.

**التحميل الكسول:** القوالب HTML تُحمّل فقط عند ظهورها في viewport، مما يقلل استهلاك الذاكرة عند وجود عدد كبير من العناصر.

**التخزين المؤقت:** استعلامات tRPC تستخدم `staleTime` الافتراضي مما يمنع إعادة الجلب المتكرر عند التنقل بين الصفحات.

**حجم البيانات:** حقل `htmlTemplate` يتراوح حجمه بين 2-8 كيلوبايت لكل عنصر، مما يجعل نقل البيانات سريعاً حتى مع عشرات العناصر.

## 10. القيود والملاحظات

1. **تخصيص القوالب محدود بالـ regex:** دالة `customizeHtmlTemplate` تعتمد على أنماط regex لاستبدال المحتوى، مما يعني أن القوالب يجب أن تتبع اصطلاحات تسمية محددة (مثل "عنوان الشريحة"، "العنوان الفرعي") ليتم استبدالها بنجاح.

2. **عدم دعم الصور في القوالب:** القوالب الحالية لا تدعم الصور المضمنة (يجب أن تكون مستقلة بالكامل). يمكن استخدام SVG مدمج كبديل.

3. **تفكيك PPTX محدود:** تحليل XML لملفات PPTX يستخرج النصوص والبنية الأساسية فقط، دون استخراج الألوان والخطوط الفعلية من التصميم. التفكيك بالذكاء الاصطناعي يعوّض ذلك جزئياً.

4. **عدم دعم الرسوم المتحركة:** القوالب الحالية ثابتة ولا تدعم الانتقالات أو الرسوم المتحركة بين الشرائح.

5. **اللغة:** النظام مصمم أساساً للغة العربية (RTL). دعم اللغة الإنجليزية متاح لكن التصميم الافتراضي يفترض اتجاه RTL.
