# Phase 9 Report

- Timestamp: 2026-03-15T01:59:06+03:00
- Phase: Excel-engine pre-implementation architecture
- Scope: Define the bounded context and shared-model strategy for Service Request 02 before any coding starts

## 1. Bounded context

- `excel-engine` owns workbook ingestion, workbook understanding, typed workbook state, formula planning/execution, structural spreadsheet transformations, formatting/style application, preview/apply orchestration, and editable workbook regeneration.
- It does not own detached storage, detached jobs, detached evidence, detached audit, detached publication, detached permissions, or detached source ingestion models.
- It consumes only the existing shared foundation packages: `contracts`, `artifacts`, `jobs`, `evidence`, `audit-lineage`, `runtime`, `library`, `permissions`, `canvas-contract`, `output-publication`, `brand-template`, and `connectors`.
- It may add workbook-specific typed schemas only inside shared packages, primarily `packages/contracts`, then re-export them through the relevant shared packages as needed.

## 3. Canonical spreadsheet mapping

- The internal editable workbook lives as `representation_kind: "spreadsheet"` in the shared canonical model and `artifact_type: "spreadsheet"` in the shared artifact model.
- Existing shared primitives already cover the top-level system-of-record shell:
  - `SheetNodeSchema` for worksheet-level node registration
  - `TableNodeSchema` and `ChartNodeSchema` for logical tabular/chart regions
  - generic `FormulaRefSchema` and `DataBindingRefSchema`
  - shared localization, layout, template refs, lineage refs, and evidence refs
- Missing workbook-native typed shared schemas that must be added before implementation:
  - workbook metadata
  - worksheet/range addressing
  - cell value/type/style state
  - merged-cell metadata
  - named-range metadata
  - table header/schema metadata
  - pivot definition/cache metadata
  - workbook diff metadata

## 4. Formula engine design

- `FormulaGraph`
  - `graph_id`
  - `workbook_ref`
  - `formula_node_refs`
  - `dependency_edge_refs`
  - `volatile_function_refs`
  - `cross_sheet_refs`
  - `recalc_strategy`
- `WorkbookFormula`
  - `formula_id`
  - `worksheet_ref`
  - `cell_ref`
  - `expression`
  - `dialect`
  - `resolved_reference_refs`
  - `dependency_refs`
  - `precedent_refs`
  - `dependent_refs`
  - `last_value_snapshot`
  - `error_state`
- Execution model:
  - parse formulas into typed AST records
  - resolve sheet/range/cell references deterministically
  - build dependency graph before apply
  - run incremental recalculation only for impacted subgraphs
  - propagate formula errors and circular-reference warnings as evidence/audit data
  - keep dynamic arrays, `LET`, and `LAMBDA` as explicit capability paths with degrade signaling if partially supported at first
- Formula outputs are not just final cell values; they must also emit lineage edges and deterministic replay metadata.

## 5. Transformation mapping

- Every structural operation must be a typed shared action definition with preview support where applicable:
  - `merge_columns`
  - `split_column`
  - `rename_column`
  - `derive_column`
  - `reorder_columns`
  - `append_table`
  - `join_tables`
  - `filter_rows`
  - `sort_range`
  - `group_aggregate`
  - `pivot_generate`
  - `unpivot_range`
  - `normalize_sheet`
  - `merge_sheets`
  - `merge_workbooks`
- Each action should produce:
  - preview artifact refs
  - transformation result metadata
  - impacted range/cell refs
  - before/after lineage edges
  - evidence checks
- Easy mode should propose safe transformations from profiling results; Advanced mode should expose join keys, target ranges, type overrides, formula controls, and sequencing explicitly.

## 6. Formatting engine plan

- Formatting must be modeled as typed workbook style metadata rather than UI-only decorations.
- Required shared workbook style surfaces:
  - cell style refs
  - row/column style refs
  - worksheet style policy
  - number/date/currency format rules
  - border/fill/font presets
  - conditional-format rules
  - auto-width heuristics metadata
  - RTL layout flags
  - freeze-pane state
- Template and style application should resolve through shared `LibraryAsset` and `TemplateBrandPreset` records.
- Formatting application remains mutating only when a preview has been accepted or an explicit direct-apply action is approved.

## 8. Output publication mapping

- Publication remains the shared `Publication` contract and should point to the workbook artifact/version/export bundle.
- Publishable outputs include:
  - editable workbook publication
  - non-editable derivative export bundle
  - downstream dataset-ready artifacts
  - dashboard/report/presentation-ready table or chart artifacts
- Editable workbook publication must stay distinct from non-editable exports so the platform does not silently downgrade editability.

## 9. Evidence / audit / lineage mapping

- Evidence must capture:
  - source files and parsed workbook stats
  - inferred table/header/type findings
  - formula parse/recalc coverage
  - transformation previews and applied mutations
  - formatting rules applied
  - pivot generation results
  - degraded behaviors and unsupported function paths
  - replay metadata for deterministic reruns
- Audit must capture:
  - requester and actor
  - top-level mode (`easy` or `advanced`)
  - AI suggestions accepted/rejected
  - explicit transformation approvals
  - formula override decisions
  - publication/export actions
- Lineage must capture:
  - source file -> workbook artifact
  - source table/range -> canonical table nodes
  - formula precedent -> dependent cell edges
  - workbook version -> next workbook version
  - transformation input refs -> output refs
  - workbook -> downstream dataset/report/presentation/dashboard artifacts

## 10. Scalability strategy

- Large workbooks require segmented parsing and background job stages:
  - ingest/parsing
  - profiling
  - planning
  - preview generation
  - apply/execution
  - verification/publication
- Recalculation must be incremental and dependency-scoped, not full-workbook only.
- Preview/apply separation is mandatory so expensive operations remain retryable and auditable.
- No hard-coded row limits should exist in contract semantics; resource profile and progress reporting should drive operational limits.

## 11. Risks

- Implementing only flat table parsing would fail the workbook, formula, formatting, and editable-output acceptance gates.
- Reusing only generic `FormulaRefSchema` without typed dependency-graph metadata would under-model recalculation determinism and error propagation.
- Pivot generation and conditional formatting can appear complete while remaining non-replayable unless typed result metadata and lineage edges are emitted.
- `excel_data` may be semantically close enough to use immediately, but it should be explicitly approved or renamed before implementation to avoid long-term contract ambiguity.

## 12. Explicit non-goals

- Macro execution for `xlsm`
- Replacing the shared source ingestion stack
- Building a detached spreadsheet viewer product
- Building a detached BI engine
- Supporting every Excel function on day one without degrade signaling
- UI-only formatting controls without typed backend state

## 13. Acceptance gates

- A real editable workbook artifact exists and points to canonical spreadsheet content.
- Workbook-native typed shared schemas are added in shared packages, not inside a private capability package.
- Formula parsing, dependency graphing, recalculation, and error propagation are real execution paths.
- Structural transformations are typed actions with preview, audit, evidence, and lineage.
- Formatting is persisted as typed style metadata and can regenerate editable workbooks.
- Pivot/table operations produce real workbook outputs and typed metadata.
- Easy and Advanced modes are explicit and never silently blended.
- Every successful or degraded execution emits evidence, audit, and lineage.
- No placeholders, parser-only flows, or viewer-only behaviors exist in the production path.

## Recommended implementation sequence after approval

1. Extend `packages/contracts` with workbook-native typed schemas and register their JSON schemas.
2. Re-export those schemas through the relevant shared packages.
3. Add excel-engine capability registration plus typed action definitions/tool registrations.
4. Implement parsing, profiling, formula graph, transformation, formatting, and workbook regeneration stages against shared job/evidence/audit/artifact contracts.
5. Add build/typecheck/lint/tests covering workbook import, formula execution, preview/apply transformations, pivot generation, formatting, and publication flows.
