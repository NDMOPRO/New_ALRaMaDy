# Phase 9 Report

- Timestamp: 2026-03-15T01:57:30+03:00
- Phase: Strict-replication-engine pre-implementation architecture
- Scope: Define the bounded context and shared-model strategy for Service Request 06 before any coding starts

## 1. Bounded context

- `strict-replication-engine` owns strict ingestion, source profiling, extraction planning, CDR absolute reconstruction, deterministic render/verify/repair execution, target exporter orchestration, round-trip validation, and strict publish eligibility decisions.
- It does not own detached artifact storage, detached jobs, detached permissions, detached evidence, detached lineage, detached publication, or detached template/brand models.
- It consumes only the existing shared foundation packages: `contracts`, `artifacts`, `jobs`, `evidence`, `audit-lineage`, `runtime`, `capability-registry`, `library`, `brand-template`, `connectors`, `permissions`, and `output-publication`.
- It must register as a first-class capability on top of `JobSchema.capability: "strict_replication"` rather than as a side workflow inside another engine.
- It may add strict-specific typed schemas only inside shared packages, primarily `packages/contracts`, then re-export them through the relevant shared packages as needed.

## 2. Strict pipeline design

- Mandatory pipeline order:
  1. source ingest + classify
  2. source fingerprint + capability matrix selection
  3. extraction plan
  4. CDR absolute build
  5. target reconstruction
  6. deterministic render
  7. Gate-1 structural equivalence
  8. Gate-2 pixel lock equivalence
  9. bounded repair loop if needed
  10. round-trip render/verify
  11. publish strict output only on full pass
- Any failure before final publish must emit evidence, audit events, lineage edges, diff artifacts, and replay metadata.
- No silent degradation is allowed in `strict`; a request that cannot pass gates must terminate as `failed` or explicit `degraded` with strict claim denied.

## 3. Source / target matrix

- Required source classes in first architecture wave:
  - PDF
  - bitmap image
  - screenshot
  - infographic image
  - dashboard/report/table image
  - OOXML document/presentation/spreadsheet where enabled
- Required strict target classes:
  - editable `PPTX`
  - editable `DOCX`
  - structured editable `XLSX`
  - live `dashboard` artifact where binding contracts can be reconstructed
- Side outputs:
  - preview PNG/PDF
  - diff artifact bundle
  - verification bundle
  - repro bundle
  - evidence pack
- Unsupported first-wave pairs must fail explicitly at planning time with evidence-backed reason codes, never downgrade silently.

## 4. CDR model plan

- The system-of-record intermediate must be a new shared typed schema family centered on `CDRAbsolute`.
- `CDRAbsolute` should extend the current canonical model rather than replace it:
  - `cdr_id`
  - `canonical_ref`
  - `source_fingerprint_ref`
  - `target_profile_ref`
  - `unit_system`
  - `layout_graph_ref`
  - `structural_hash`
  - `pixel_hash_seed`
  - `deterministic_profile_ref`
  - `snapshot_version_ref`
- Required strict node families:
  - container/group nodes
  - frame/page/sheet/slide nodes
  - text run nodes with glyph metrics and bidi metadata
  - vector primitive nodes
  - image nodes with crop/mask metadata
  - table/grid nodes with span/formula/style metadata
  - chart nodes with series/axis/style/binding metadata
  - filter/control nodes for live dashboard reconstruction
- Required strict metadata:
  - exact dimensions and units
  - transforms
  - z-index
  - typography metrics
  - sRGB lock
  - structural hashes by subtree
  - editability locks
  - extraction confidence annotations

## 5. Extraction model plan

- Shared strict extraction schemas should capture:
  - `SourceFingerprint`
  - `ExtractionManifest`
  - `FontMetricFingerprint`
  - `VisualObjectInventory`
  - `TableStructureExtraction`
  - `ChartStructureExtraction`
  - `OCRSpanMap`
  - `BindingRecoveryPlan`
- Extraction paths:
  - PDF vector/text path where available
  - OOXML native structure path where available
  - bitmap/OCR path
  - icon/vector segmentation path
  - table detection path
  - chart detection path
  - font recognition + metric measurement path
- Arabic support requirements:
  - bidi-aware tokenization
  - shaping-aware text runs
  - line-break parity evidence
  - no silent font fallback
  - no text-to-image substitution in editable outputs

## 6. Deterministic render environment plan

- STRICT requires a dedicated isolated execution path with immutable runtime configuration.
- The deterministic profile should include:
  - renderer identifiers and versions
  - font set fingerprint
  - locale and numeral system
  - color profile fixed to sRGB
  - floating-point normalization rules
  - seed value
  - DPI and viewport rules
  - antialiasing policy
  - OS/container stamp
- The environment stamp must be recorded in the evidence pack and referenced by every render stage.
- Runtime mutation is forbidden once a strict job enters execution; configuration drift invalidates the run.

## 7. Dual gate model

- Gate-1 `STRUCTURAL_EQ` validates:
  - layout graph equivalence
  - element inventory parity
  - hierarchy and grouping parity
  - text run order and style preservation
  - table/chart structure parity
  - formula and live-binding parity where applicable
- Gate-2 `PIXEL_LOCK_EQ` validates:
  - deterministic render of source baseline vs reconstructed target render
  - exact `PixelDiff == 0.0`
  - identical render fingerprint inputs
- Shared verification schemas should model:
  - `StructuralGateResult`
  - `PixelGateResult`
  - `DualGateSummary`
  - `RoundTripGateResult`
- A strict publish claim is valid only when all gates pass, including post-export round-trip validation.

## 8. Repair / degrade policy

- Repair loop is bounded and evidence-backed:
  - render
  - compare
  - identify mismatch class
  - apply deterministic repair recipe
  - rerender
  - reverify
- Shared repair schemas should capture:
  - `RepairIteration`
  - `RepairAction`
  - `RepairOutcome`
  - `MismatchClassification`
- Allowed repair classes:
  - font mapping correction
  - wrap/line-break correction
  - coordinate rounding correction
  - vector primitive normalization
  - table grid reconstruction correction
  - chart layout correction
- Degrade policy:
  - no silent degrade in strict mode
  - explicit `degraded` state only with reason codes and blocked strict publish
  - if editability would be flattened, the run must fail strict rather than emit a fake editable output

## 9. Round-trip validation model

- Every exported strict target must be rendered again under the deterministic environment and compared back against the accepted reconstruction baseline.
- Round-trip validation should capture:
  - export artifact ref
  - re-render artifact ref
  - structural and pixel gate summaries
  - regression diff refs
  - functional checks by target type
- Functional checks by target class:
  - `PPTX`: editable text/layout/theme integrity
  - `DOCX`: editable sections, tables, charts, and text-flow integrity
  - `XLSX`: workbook structure, panes, formulas, and cell formatting integrity
  - `dashboard`: bindings, filters, drill actions, and KPI/chart parity

## 10. Artifact mapping

- Source ingest: shared `Artifact`
  - `artifact_type: "source_file"`
- CDR absolute snapshot: shared `Artifact`
  - `artifact_type: "workflow_output"`
  - `artifact_subtype: "cdr_absolute_snapshot"`
- Strict editable output by family:
  - presentation target: `artifact_type: "presentation"`
  - report target: `artifact_type: "report"`
  - spreadsheet target: `artifact_type: "spreadsheet"`
  - dashboard target: `artifact_type: "dashboard"`
- Strict publish eligibility bundle:
  - `artifact_type: "strict_output"`
  - `artifact_subtype: "verified_strict_output"`
- Preview renders:
  - `artifact_type: "preview_render"`
- Export bundle:
  - `artifact_type: "export_bundle"`
- Verification and repro bundles:
  - `artifact_type: "evidence_pack"`
  - `artifact_subtype: "strict_verification_bundle"` or `repro_bundle`

## 11. Evidence / audit / lineage mapping

- Evidence must capture:
  - source fingerprints
  - extraction manifests
  - CDR snapshot refs
  - deterministic environment stamp
  - renderer versions
  - gate results
  - every repair iteration
  - diff artifacts
  - round-trip results
  - failure/degrade reason codes
- Audit must capture:
  - requester
  - mode (`easy` or `advanced`)
  - source/target override choices
  - strict profile selection
  - repair approvals if any policy requires them
  - publish/block decisions
  - explicit denial of strict claims when a gate fails
- Lineage must capture:
  - source artifact -> extraction manifest
  - extraction manifest -> CDR snapshot
  - CDR snapshot -> reconstructed target artifact
  - target artifact -> verification bundle
  - target artifact -> export bundle
  - target artifact -> round-trip rerender
  - failure run -> repro bundle

## 12. Risks

- The current shared contracts do not yet contain typed strict schemas; implementing the engine before adding them would force private incompatible models.
- Pixel parity can become non-repeatable if font sets, rasterization policy, locale, or floating-point behavior drift between stages.
- Arabic shaping and font-metric parity are likely the hardest acceptance gate because screenshot/image sources do not preserve original semantic text runs.
- Live dashboard strict output is only valid when binding recovery is explicit and verifiable; otherwise the run must fail strict instead of flattening interactivity.

## 13. Explicit non-goals

- Building a best-effort converter and calling it strict
- Publishing screenshot-based outputs as editable strict targets
- Allowing approximate pixel diffs in strict mode
- Bypassing CDR absolute with direct format-to-format shortcuts
- Silent font substitution
- Flattened editable targets
- Detached verification outside shared evidence/audit/lineage

## 14. Acceptance gates

- A real strict pipeline exists and enforces the mandatory stage order.
- Shared strict schemas for CDR, extraction, gate results, repair traces, and round-trip validation are added inside shared packages.
- A deterministic isolated render environment exists and is recorded in evidence.
- Gate-1 structural equivalence exists and is enforced.
- Gate-2 pixel lock equivalence exists and requires exact `PixelDiff == 0.0`.
- Bounded repair loop exists with evidence for each iteration.
- Editable target outputs are real typed artifacts, not flattened screenshots.
- Round-trip validation exists before strict publish eligibility is granted.
- Failure and degrade paths always emit evidence, audit, lineage, diff artifacts, and repro metadata.
- No docs-only or mock-only path is presented as completion.

## Recommended implementation sequence after approval

1. Extend `packages/contracts` with typed strict schemas for CDR, extraction, deterministic render profiles, dual-gate results, repair traces, and round-trip validation.
2. Re-export those schemas through the relevant shared packages and register JSON schemas.
3. Add strict capability registration, action manifests, tool registrations, and evidence hooks.
4. Implement the isolated deterministic execution path and stage orchestrator on top of shared jobs/artifacts/evidence/audit contracts.
5. Implement source adapters, target exporters, dual-gate verification, bounded repair loop, and round-trip validation.
6. Add build/typecheck/lint/tests covering strict pass, strict fail, degrade-denied publish, Arabic typography parity, and repro bundle generation.
