# Phase 7 Report

- Timestamp: 2026-03-15T01:47:40+03:00
- Phase: Report-engine pre-implementation architecture
- Scope: Define the bounded context and shared-model strategy for Service Request 04 before any coding starts

## 1. Bounded context

- `report-engine` owns report planning, composition, binding, versioning, scheduling, review/publish orchestration, export orchestration, and derived conversion orchestration.
- It does not own detached storage, detached permissions, detached publication, detached evidence, detached canvas UX, or detached artifact/job models.
- It consumes only the existing shared foundation packages: `contracts`, `artifacts`, `jobs`, `evidence`, `audit-lineage`, `library`, `permissions`, `canvas-contract`, `runtime`, `output-publication`, and `brand-template`.
- It may add report-specific typed schemas only inside shared packages, primarily `packages/contracts`, then re-export them through the relevant shared packages as needed.

## 3. Section / layout / content block model

- The internal editable report lives as `representation_kind: "report"` in the shared canonical model.
- Report structure is represented as canonical document/page/text/table/chart/image nodes plus report-specific shared metadata:
  - section tree
  - block ordering
  - page/appendix anchors
  - TOC anchors
  - citation anchors
  - lock/editability rules
- Layout rules stay template-aware through `template_refs`, `layout_ref`, `layout_metadata`, and shared brand presets.
- Drag-and-drop applies to:
  - source artifact to section
  - dataset binding to chart/table block
  - library asset/template to report
  - section reorder inside the canvas session

## 4. Data binding model

- `ReportBindingSet`
  - `binding_set_id`
  - `source_artifact_refs`
  - `dataset_binding_refs`
  - `staleness_status`
  - `refresh_policy`
  - `selective_regeneration_refs`
  - `broken_binding_refs`
- Each bound table/chart/metric block keeps:
  - `binding_id`
  - `dataset_ref`
  - `query_ref`
  - `target_node_ref`
  - `field_mappings`
  - `snapshot_version_ref`
  - `last_refresh_at`
- Narrative blocks that summarize data carry:
  - source refs
  - accepted AI suggestion refs if AI-assisted
  - a `derived_from_binding_refs` list for traceability
- A report can be:
  - fully live-bound
  - partially live-bound
  - frozen snapshot
- “live” is valid only when staleness detection and source-health checks pass.

## 5. Scheduling / publication model

- `ReportSchedule`
  - `schedule_id`
  - `report_ref`
  - `cadence` (`weekly`, `monthly`, `on_demand`)
  - `calendar_rule`
  - `publication_target_refs`
  - `next_run_at`
  - `last_run_at`
  - `active`
  - `job_template_ref`
- Scheduled executions instantiate shared `Job` records with `capability: "reports"`.
- Publication remains a shared `Publication` record pointing to the report artifact/version artifact/export bundle.
- Official/printable output metadata includes:
  - legal-ready flag
  - printable flag
  - scheduled publication refs
  - visibility scope

## 6. Comparison / versioning model

- `ReportVersion` is the authoritative version chain.
- `ReportDiff`
  - `diff_id`
  - `base_version_ref`
  - `target_version_ref`
  - `compare_kind` (`version_to_version`, `period_to_period`, `dataset_to_dataset`)
  - `changed_section_refs`
  - `changed_table_refs`
  - `changed_chart_refs`
  - `added_refs`
  - `removed_refs`
  - `changed_metric_refs`
  - `summary_ref`
- Diff results produce:
  - diff report artifact
  - lineage edges with `version_diff_ref`
  - audit events for the compare action

## 7. Template integration model

- Report templates are library assets with `asset_type: "template"` and report-specific shared metadata.
- Brand and layout rules come only from `TemplateBrandPreset`.
- Template resolution inputs:
  - report type
  - audience
  - mode
  - workspace/org preset
  - source profile
  - regulatory/comparison/trend flags
- Template lock semantics:
  - `soft_lock`: editable but tracked override
  - `strict_lock`: locked sections/layout regions except approved override actions
- External report pattern import creates:
  - extracted style/layout hints
  - reusable template candidate
  - warnings/evidence for degraded extraction when structure cannot be fully reconstructed

- Derived dashboard conversion: shared `Artifact`

- Derived presentation conversion: shared `Artifact`

- Evidence pack: shared `Artifact`

## 9. Evidence / audit / lineage mapping

- Evidence must capture:
  - executed pipeline stages
  - sources used
  - template chosen
  - sections created/regenerated
  - bound tables/charts/metrics
  - refresh, compare, review, approval, publication, and schedule actions
  - degraded extraction or broken bindings
  - reproducibility metadata
- Audit must capture:
  - requester
  - top-level mode (`easy` or `advanced`)
  - AI suggestions accepted/rejected
  - review comments and approval decisions
  - export/publication actions
  - schedule create/update/disable actions
  - template overrides and lock bypass approvals
- Lineage must capture:
  - source artifact -> report artifact
  - source dataset binding -> block nodes
  - report version -> next report version
  - report -> diff artifact
  - report -> dashboard
  - report -> presentation
  - template -> report version
  - schedule -> generated report instances

## 10. Risks

- The current shared contracts do not yet contain typed report-specific schemas; implementing the service before adding them would force private models and violate the request.
- External PDF/scan pattern import may produce only partial structure; degraded behavior must stay explicit and evidenced.
- Live refresh without per-block binding lineage will fail acceptance for stale data and selective regeneration.
- Review/publish can become superficial if approval state is stored only in UI state instead of versioned shared metadata plus audit events.

## 11. Explicit non-goals

- Building Presentation Engine itself
- Building Strict Engine itself
- Building a standalone rich-text editor product
- Building an email platform
- Building broad new connectors outside current shared foundation
- Building advanced multi-user collaborative authoring beyond tracked review/approval

## 12. Acceptance gates

- Report-specific shared schemas are added inside shared packages, not in a private report package.
- A real editable internal report artifact exists and points to canonical report content.
- DOCX/PDF/HTML are derived outputs, not the only persistence path.
- Easy and Advanced are explicit and never blended silently.
- Live refresh and stale-binding detection are version-aware.
- Weekly/monthly/on-demand scheduling materializes real jobs and publication links.
- Draft -> review -> approval -> publish is tracked with timestamps, actors, comments, and evidence hooks.
- Comparison emits a diff artifact plus table/chart/section change metadata.
- Templates and important report versions publish into the library with dependency refs.
- Every successful or degraded execution emits evidence, audit, and lineage.
- No placeholders, mocks, or detached mini-service behaviors exist in the production path.

## Recommended implementation sequence after approval

1. Extend `packages/contracts` with typed report schemas and register their JSON schemas.
2. Re-export those schemas through the appropriate shared packages.
3. Add report capability registration, action manifests, and tool registrations.
4. Implement the pipeline stages against shared job/evidence/audit/artifact contracts.
5. Add build/typecheck/lint/tests covering report creation, refresh, compare, schedule, review, and publication flows.
