# Phase 9 Report

- Timestamp: 2026-03-15T01:58:13+03:00
- Phase: Presentation/infographic-engine pre-implementation architecture
- Scope: Define the bounded context and shared-model strategy for Service Request 05 before any coding starts

## 1. Bounded context

- `presentation-infographic-engine` owns intent capture, outline generation, storyboard planning, deck composition, infographic composition, slide-level mutation, data binding inside slides, speaker notes generation, preview/render parity validation, and export orchestration.
- It does not own detached storage, detached permissions, detached publication, detached evidence, detached artifact/job models, or a standalone presentation editor product.
- It consumes only the existing shared foundation packages: `contracts`, `artifacts`, `jobs`, `evidence`, `audit-lineage`, `library`, `permissions`, `canvas-contract`, `runtime`, `output-publication`, and `brand-template`.
- It may add presentation-specific typed schemas only inside shared packages, primarily `packages/contracts`, then re-export them through the relevant shared packages as needed.

## 5. PPTX object model plan

- The internal editable deck artifact is the system of record; PPTX is a derived export that must preserve semantic editability rather than flattening into images.
- Mapping rules:
  - text nodes -> editable PPTX text boxes/runs
  - chart nodes -> editable PPTX charts with binding metadata
  - table nodes -> editable PPTX tables
  - image/icon/media nodes -> structured media placements
  - layout zones/master refs -> PPTX layouts, masters, and placeholders where applicable
  - speaker notes -> PPTX notes pages
- No screenshot-only path is valid in the editable export flow.
- If any source-to-slide transformation cannot preserve semantic editability, the engine must degrade explicitly, record the reason, and block any “premium editable” claim.

## 6. Template / brand integration model

- Presentation templates remain shared library assets with `asset_type: "template"` or `asset_type: "presentation"` plus typed presentation metadata.
- Brand and layout rules come only from `TemplateBrandPreset`.
- Template resolution inputs:
  - deck type
  - audience
  - tone
  - language
  - mode
  - workspace/org preset
  - source profile
  - rtl requirements
- Template lock semantics:
  - `soft_lock`: editable but tracked override
  - `strict_lock`: locked zones, styles, or motion rules except via approved override actions
- “match another deck design” should resolve through extracted theme/layout hints stored as shared metadata and evidenced if the reconstruction is partial.

## 7. Preview / export / parity validation model

- Preview rendering produces shared `preview_render` artifacts and campus reader publications.
- Export orchestration produces a shared export bundle containing PPTX/PDF/HTML derivatives.
- `RenderParityValidation`
  - `parity_validation_id`
  - `deck_version_ref`
  - `preview_ref`
  - `pptx_export_ref`
  - `pdf_export_ref`
  - `html_export_ref`
  - `checks`
  - `severity`
  - `auto_fix_attempted`
  - `degrade_reason`
- Mandatory parity checks:
  - overflow/clipping
  - Arabic/RTL layout direction
  - table and chart editability
  - notes presence
  - brand/theme consistency
  - template-lock compliance
  - reader/PPTX/PDF/HTML structural equivalence

## 9. Evidence / audit / lineage mapping

- Evidence must capture:
  - sources used
  - chosen template/theme/brand preset
  - intent manifest, outline, and storyboard refs
  - generated or regenerated slides
  - infographic/chart/table generation details
  - parity checks and auto-fix attempts
  - degraded editability, overflow, or partial reconstruction warnings
  - reproducibility metadata for export and preview runs
- Audit must capture:
  - requester
  - top-level mode (`easy` or `advanced`)
  - AI suggestions accepted/rejected
  - slide add/delete/reorder/regenerate actions
  - template overrides and lock bypass approvals
  - publication and export actions
- Lineage must capture:
  - source artifact -> deck artifact
  - intent -> outline -> storyboard -> deck version
  - dataset binding -> slide block
  - template/theme -> deck version
  - deck version -> parity validation artifact
  - deck version -> export bundle

## 10. Risks

- The current shared contracts do not yet contain typed presentation-specific schemas; implementing the service before adding them would force private models and violate the request.
- The canonical node inventory does not yet model speaker notes, video/media blocks, motion metadata, or grouped infographic semantics as first-class typed structures.
- PPTX parity can appear complete while still failing acceptance if text, charts, or tables are flattened during export.
- Arabic/RTL support can fail in notes, overflow handling, or theme locks even when the preview appears visually correct.

## 11. Explicit non-goals

- Building a standalone PowerPoint replacement product
- Building a video-editing suite
- Building detached template or media stores outside the shared foundation
- Building fake motion or premium export claims without parity evidence
- Building broad collaborative editing beyond tracked review/approval operations

## 12. Acceptance gates

- Real shared `PresentationIntentManifest`, outline, and storyboard models exist.
- Presentation-specific shared schemas are added inside shared packages, not in a private presentation package.
- A real editable internal deck artifact exists and points to canonical presentation content.
- PPTX export preserves text as text, charts as charts, and tables as editable tables.
- Slide-by-slide add/delete/reorder/regenerate operations are version-aware and evidence-backed.
- Data bindings survive compatible block replacements and are explicitly degraded otherwise.
- Templates, brand presets, lock rules, and RTL behavior are enforced and auditable.
- Campus reader preview exists and parity validation covers preview, PPTX, PDF, and HTML.
- Every successful or degraded execution emits evidence, audit, and lineage.
- No screenshot-only editable path, mock template path, or detached mini-service behavior exists in production.

## Recommended implementation sequence after approval

1. Extend `packages/contracts` with typed presentation schemas for intent, outline/storyboard, deck metadata, slide/block metadata, notes, motion, and parity validation.
2. Re-export those schemas through the appropriate shared packages.
3. Add presentation capability registration, action manifests, and tool registrations.
4. Implement generation, edit, regenerate, binding-refresh, parity, and export stages against shared job/evidence/audit/artifact contracts.
5. Add build/typecheck/lint/tests covering prompt-to-deck, source-to-deck, slide mutation, infographic replacement, RTL, parity validation, and editable PPTX export.
