# Phase 10 Report

- Timestamp: 2026-03-15T02:01:05+03:00
- Phase: LCT engine pre-implementation architecture
- Scope: Define the bounded context and shared-model strategy for Service Request 07 before any coding starts

## 1. Bounded context

- `arabic-localization-lct-engine` owns language transformation, terminology application, direction/layout transformation planning, typography refinement, cultural formatting, localization preview/apply orchestration, quality verification, and repair/degrade orchestration.
- It does not own detached artifact storage, detached document models, detached permissions, detached evidence/audit/lineage, detached publication, or private layout models outside shared packages.
- It consumes the existing shared foundation packages: `contracts`, `artifacts`, `jobs`, `evidence`, `audit-lineage`, `library`, `permissions`, `runtime`, `output-publication`, and `brand-template`.
- It should register itself as the shared `JobSchema` capability `lct`, not as a detached mini-service.
- It may add LCT-specific typed schemas only inside shared packages, primarily `packages/contracts`, then re-export them through the relevant shared packages as needed.

## 3. Terminology / profile model

- `TerminologyProfile`
  - `profile_id`
  - `scope` (`org`, `workspace`, `domain`, `user`, `artifact_override`)
  - `source_locale`
  - `target_locale`
  - `default_style` (`executive`, `formal`, `technical`)
  - `strictness_mode` (`strict_localization`, `smart_localization`, `pro_localization`)
  - `brand_preset_ref`
  - `rule_refs`
- `TerminologyRule`
  - `rule_id`
  - `term`
  - `preferred_translation`
  - `banned_translation_refs`
  - `protected`
  - `case_policy`
  - `context_notes`
- Profile overlays must resolve in this order:
  - org
  - workspace
  - domain
  - user
  - artifact override
- Protected terms, title/caption policy, acronym policy, and non-translatable spans must be represented explicitly and emitted in evidence.

## 4. RTL / LTR transformation model

- The editable source and target live as shared `CanonicalRepresentation` records.
- LCT must operate as canonical graph transformation, not string replacement.
- `DirectionalLayoutPlan`
  - `plan_id`
  - `source_direction`
  - `target_direction`
  - `affected_node_refs`
  - `table_transform_refs`
  - `chart_transform_refs`
  - `widget_transform_refs`
  - `mirror_policy`
  - `reading_flow_policy`
  - `overflow_risk_refs`
- Each affected node keeps transformation metadata for:
  - text alignment
  - reading order
  - padding/margin swaps
  - anchor flips
  - axis/legend relocation
  - cell order rules when semantically valid
- Mixed-direction sections must be handled per node/span and never by forcing a document-global mirror pass.

## 5. Typography refinement model

- `TypographyRefinementPlan`
  - `typography_plan_id`
  - `font_policy_ref`
  - `line_height_policy`
  - `baseline_mixing_policy`
  - `numeric_shaping_policy`
  - `fallback_font_refs`
  - `clipping_repair_refs`
- Typography refinement must cover:
  - Arabic shaping correctness
  - line-height expansion/compression
  - mixed Arabic/Latin baseline correction
  - fallback font policy
  - headline/body/caption hierarchy preservation
- Font substitution must remain explicit and evidenced; silent substitution is not acceptable.

## 6. Cultural formatting model

- `CulturalFormattingPlan`
  - `format_plan_id`
  - `date_policy`
  - `time_policy`
  - `number_policy`
  - `currency_policy`
  - `separator_policy`
  - `locale_phrase_policy`
  - `asset_direction_policy`
- The plan must support:
  - locale-specific dates and times
  - number separators and decimal style
  - currency symbol placement
  - optional culturally adapted phrasing when policy allows
  - icon/arrow/asset direction adaptation where semantically necessary

## 7. Quality gate model

- `LocalizationQualityReport`
  - `quality_report_id`
  - `language_quality`
  - `layout_quality`
  - `editability_quality`
  - `cultural_quality`
  - `overall_outcome`
  - `repair_recommendation_refs`
- Required quality families:
  - `LANGUAGE_QUALITY`
  - `LAYOUT_QUALITY`
  - `EDITABILITY_QUALITY`
  - `CULTURAL_QUALITY`
- Each quality result should map onto shared `ValidationCheckResultSchema` entries with impacted node/artifact refs.
- Quality failures that can be repaired should emit structured repair actions; non-repairable failures must emit degrade reasons and block apply under strict policy.

## 8. Repair / degrade policy model

- `LocalizationDegradePolicy`
  - `policy_id`
  - `strict_apply_behavior`
  - `smart_apply_behavior`
  - `preview_blocking_threshold`
  - `repair_first_rules`
  - `non_editable_block_rule`
- Outcomes must map to shared degrade/evidence states:
  - `success`
  - `success_with_warnings`
  - `degraded`
  - `failed`
- Typical degrade triggers:
  - unresolved protected-term conflict
  - chart label overflow not repairable
  - non-editable text flattening risk
  - unsupported mixed-direction layout region
  - missing Arabic-capable fallback font

## 9. Artifact mapping

- Source editable artifact: shared `Artifact`
  - `artifact_type`: one of `report`, `presentation`, `dashboard`, `spreadsheet`
  - `canonical_ref` -> source editable canonical
- Preview localization output: shared `Artifact`
  - `artifact_type: "preview_render"`
  - references preview canonical diff or render
- Localized editable artifact: shared `Artifact`
  - same first-class artifact type as the source category
  - updated `canonical_ref` -> localized editable canonical
- Localization diff artifact: shared `Artifact`
  - `artifact_subtype: "localization_diff"`
- Quality verification bundle: shared `Artifact`
  - `artifact_type: "evidence_pack"`
- Export bundle: shared `Artifact`
  - `artifact_type: "export_bundle"`
  - contains PDF/HTML/PPTX/XLSX derivatives when needed
- Terminology application report can be:
  - evidence attachment
  - diff artifact metadata
  - library asset when promoted as a reusable profile report

## 10. Evidence / audit / lineage mapping

- Evidence must capture:
  - source and target locales
  - terminology profile applied
  - protected/non-translatable term decisions
  - directional/layout transforms executed
  - typography refinements applied
  - cultural formatting rules applied
  - quality checks and repair attempts
  - preview/apply decision path
  - degrade reasons and editability coverage
- Audit must capture:
  - requester
  - top-level mode (`easy` or `advanced`)
  - policy type used (`strict`, `smart`, `pro`)
  - approval state
  - manual overrides
  - glossary/profile changes
  - non-translatable/protected term overrides
  - publish/export actions
- Lineage must capture:
  - source artifact -> localized artifact
  - source canonical node -> localized canonical node
  - terminology profile -> localization version
  - brand preset/template -> localized output
  - preview artifact -> approved apply artifact
  - repair pass -> final artifact
  - diff artifact -> compared versions

## 11. Risks

- The current shared contracts do not yet contain typed LCT-specific schemas; implementing the service before adding them would force private models and violate the request.
- Naive mirroring can break semantic layout even when text translation quality is high.
- Editable output preservation will fail if any transformation path rasterizes text or detaches live bindings.
- Mixed Arabic/English content needs per-span handling; block-level translation alone is insufficient.
- Font fallback gaps can make Arabic shaping appear correct in some views while failing in exports or downstream editing tools.

## 12. Explicit non-goals

- Building a generic machine-translation-only service
- Building a detached document editor
- Replacing report/presentation/dashboard/spreadsheet capabilities with private localized copies
- Silent auto-apply without preview/evidence
- Flattening outputs into image-only artifacts
- Broad new connector work outside current shared foundation

## 13. Acceptance gates

- LCT-specific shared schemas are added inside shared packages, not in a private capability package.
- A real preview-before-apply path exists and is enforced through shared action/artifact/evidence primitives.
- Easy and Advanced are explicit and never blended silently.
- Strict, Smart, and Pro localization policies are explicitly modeled and never conflated.
- Direction transformation, typography refinement, and cultural formatting each have first-class modeled execution paths.
- Editable outputs remain editable and retain hierarchy and bindings where the source was editable.
- Quality gates produce real structured results and can block apply when policy requires.
- Every successful or degraded execution emits evidence, audit events, and lineage edges.
- Diff and preview artifacts exist for inspection before mutation.
- No placeholders, mocks, or string-only translation paths exist in the production path.

## Recommended implementation sequence after approval

1. Extend `packages/contracts` with typed LCT schemas and register their JSON schemas.
2. Re-export those schemas through the relevant shared packages.
3. Add LCT capability registration, action manifests, and tool registrations.
4. Implement preview, apply, verify, and repair pipeline stages against shared job/action/evidence/audit/artifact contracts.
5. Add build/typecheck/lint/tests covering translation, direction change, typography, cultural formatting, quality gating, preview/apply, and degrade flows.
