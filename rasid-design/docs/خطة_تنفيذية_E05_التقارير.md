# الخطة التنفيذية الشاملة — محرك التقارير (E05)

## ملخص تنفيذي
يهدف محرك التقارير (E05) إلى توفير منظومة متكاملة لإنشاء تقارير احترافية عالية الجودة، تدعم اللغة العربية بشكل متقدم (Arabic ELITE)، وتتوافق مع متطلبات الحوكمة الصارمة في البيئات الحكومية والتجارية. يقوم المحرك بتحويل المدخلات المتنوعة (نصوص، بيانات، ملفات) إلى مستندات DOCX قابلة للتحرير الكامل، مع إمكانية تصديرها إلى صيغ PDF و HTML. يتميز المحرك بقدرته على التكامل مع مصادر البيانات المختلفة لإنشاء محتوى مرتبط بالبيانات (Data-Bound)، ويدعم قوالب متعددة تشمل التصميم وأسلوب الكتابة معاً. من أهم مبادئه الأساسية منع اختلاق البيانات أو الادعاءات غير المثبتة، حيث يُلزم بتقديم حزمة أدلة (Evidence Pack) مع كل تقرير مُصدَّر لضمان الشفافية والموثوقية. يعمل المحرك ضمن واجهة استخدام موحدة (Single Canvas) لتبسيط تجربة المستخدم، ويتبع نهج التحكم التدريجي (Progressive Controls) لعرض الخيارات ذات الصلة بالسياق الحالي فقط، مما يمنع إرباك المستخدم. تم تصميم المحرك ليكون المرجع الوحيد لتنفيذ هذه المتطلبات، وأي سلوك غير منصوص عليه يُعتبر ممنوعاً بشكل قاطع.

## أولاً: المتطلبات المنفّذة بالكامل ✅
*لا يوجد متطلبات منفذة حالياً.*

## ثانياً: المتطلبات المنفّذة جزئياً 🟡 أو الوهمية ⚠️
*لا يوجد متطلبات منفذة جزئياً أو وهمية حالياً. جميع المتطلبات سيتم تنفيذها من خلال الخطة أدناه.*

## ثالثاً: الخطة التنفيذية المرحلية

### المرحلة 1: بناء التمثيل الوسيط للمستند (DOC-IR)

#### المهمة 1.1: تصميم وتنفيذ الهيكل الأساسي لـ DOC-IR
- **المتطلبات المشمولة**:
  - `E05-0075`: المنفّذ MUST يبني DOC-IR لتوحيد البناء/التحقق/التصدير ومنع الاجتهاد.
  - `E05-0076`: locale, arabic_mode
  - `E05-0077`: page_setup (paper_size, margins, rtl)
  - `E05-0078`: template_refs (brand, report, writing)
  - `E05-0079`: sections[] (ordered)
  - `E05-0080`: global_fields (date, author, classification, recipients)
  - `E05-0081`: references[] (optional)
  - `E05-0082`: data_bindings[] (tables/charts/kpis)
  - `E05-0083`: fingerprints: layout_hash, style_hash, writing_hash, binding_hash
- **الحالة الحالية**: غير منفّذ
- **خطوات التنفيذ**:
  1. تصميم مخطط JSON Schema لـ DOC-IR يغطي كافة المتطلبات من E05-0076 إلى E05-0083.
  2. كتابة كود لإنشاء والتحقق من صحة كائنات DOC-IR بناءً على المخطط.
  3. تنفيذ وظائف لحساب التجزئة (fingerprints) لمكونات التخطيط، الأنماط، الكتابة، وربط البيانات.

#### المهمة 1.2: تعريف مكونات الأقسام (Sections) والمكعبات (Blocks)
- **المتطلبات المشمولة**:
  - `E05-0084`: section_id, index, title
  - `E05-0085`: kind: cover\|toc\|executive_summary\|body\|findings\|recommendations\|appendix\|glossary\|references\|signoff
  - `E05-0086`: blocks[] ordered
  - `E05-0087`: header/footer overrides (optional)
  - `E05-0088`: numbering scheme (optional)
  - `E05-0089`: kind: heading\|paragraph\|bullets\|table\|chart\|kpi_cards\|figure\|callout\|quote\|code\|appendix_table\|signature\|page_break
  - `E05-0090`: content (typed)
  - `E05-0091`: data_binding_ref (optional)
  - `E05-0092`: rtl_policy (auto/force_rtl)
- **الحالة الحالية**: غير منفّذ
- **خطوات التنفيذ**:
  1. توسيع مخطط JSON Schema ليشمل تعريفات `Section` و `Block`.
  2. تنفيذ أنواع `Block` المختلفة كاتحاد (Union) ضمن المخطط، مع تحديد المحتوى المتوقع لكل نوع.
  3. تطوير منطق لترتيب المكعبات داخل الأقسام والحفاظ على تسلسلها.
  4. إضافة دعم للخيارات الإضافية في الأقسام مثل `header/footer overrides` و `numbering scheme`.

### المرحلة 2: تطوير واجهة المستخدم وتجربة الاستخدام

#### المهمة 2.1: بناء واجهة التحرير الرئيسية (Report Canvas & Side Panel)
- **المتطلبات المشمولة**:
  - `E05-0012`: **Canvas واحد**: إنشاء/تعديل التقرير داخل واجهة واحدة + Panel واحد قابل للإخفاء (لا صفحات كثيرة).
  - `E05-0020`: **Report Canvas**: محرر مستند + Outline navigator + blocks.
  - `E05-0021`: **Side Panel واحد** (قابل للإخفاء) يضم: Library, Styles & Templates, Data Picker, Blocks, Citations & References, Governance, Export, History/Versions.
- **الحالة الحالية**: غير منفّذ
- **خطوات التنفيذ**:
  1. تطوير مكون `Report Canvas` باستخدام مكتبة تحرير نصوص غنية تدعم المكعبات (Blocks).
  2. بناء مكون `Outline Navigator` الذي يعكس هيكل المستند (الأقسام والعناوين) بشكل شجري.
  3. تصميم وتنفيذ `Side Panel` كحاوية واحدة قابلة للإخفاء، تحتوي على تابات أو أقسام للمكونات الفرعية المذكورة في `E05-0021`.
  4. ربط `Outline Navigator` و `Side Panel` مع `Report Canvas` لضمان التفاعل المتزامن بينها.

#### المهمة 2.2: تطبيق آلية التحكم التدريجي (Progressive Controls)
- **المتطلبات المشمولة**:
  - `E05-0022`: MUST عدم عرض خيارات كثيرة دفعة واحدة.
  - `E05-0023`: MUST عرض 5–9 خيارات حسب السياق: report-level, section, element controls.
  - `E05-0024`: MUST توفير “Search Controls” للبحث عن أي خيار بدل قوائم ضخمة.
- **الحالة الحالية**: غير منفّذ
- **خطوات التنفيذ**:
  1. تطوير مدير سياق (Context Manager) يتتبع العنصر المحدد حالياً في الواجهة (التقرير، قسم، فقرة، جدول).
  2. بناء منطق لعرض لوحات تحكم (Control Panels) مختلفة في `Side Panel` بناءً على السياق الذي يوفره المدير.
  3. لكل لوحة تحكم، يتم تصميمها لعرض الخيارات الأكثر أهمية فقط (5-9 خيارات).
  4. إضافة حقل بحث (`Search Controls`) في أعلى `Side Panel` للوصول السريع إلى أي خاصية أو إعداد.

#### المهمة 2.3: ضمان المعاينة الفورية ومنع تجميد الواجهة
- **المتطلبات المشمولة**:
  - `E05-0025`: أي تعديل كبير MUST ينتج: Preview سريع داخل canvas و Preview “Reader” (PDF/HTML) عند الطلب.
  - `E05-0026`: لا تجميد واجهة. أي render ثقيل MUST job بالخلفية.
- **الحالة الحالية**: غير منفّذ
- **خطوات التنفيذ**:
  1. تنفيذ آلية معاينة سريعة داخل `Canvas` تعكس التغييرات على التخطيط والأنماط بشكل فوري.
  2. تطوير خدمة خلفية (Background Job) لتوليد معاينة عالية الدقة (PDF/HTML) عند الطلب.
  3. ربط الخدمة الخلفية بالواجهة لعرض مؤشر تحميل (loading indicator) ومن ثم عرض المعاينة في نافذة منبثقة (overlay) عند جهوزيتها.

### المرحلة 3: منطق التوليد الأساسي وأنماط المحتوى

#### المهمة 3.1: تطبيق تدفقات التوليد (AUTO و CONTROLLED)
- **المتطلبات المشمولة**:
  - `E05-0027`: من Prompt واحد (أو مع مصادر بيانات) يُنتج التقرير.
  - `E05-0028`: Intent Manifest → Outline → Narrative Plan → Data Bindings → DOCX Build → QA → Export/Evidence
  - `E05-0029`: بدون أسئلة.
  - `E05-0030`: المستخدم يضبط knobs قبل التنفيذ (في Panel).
  - `E05-0031` - `E05-0039`: (report_type, language, tone, fidelity_mode, template_id, add_sections, citation_mode, sensitivity_classification, export_targets)
  - `E05-0032`: language (ar/en/mixed)
  - `E05-0033`: tone (formal/neutral/persuasive/urgent)
  - `E05-0034`: fidelity_mode (literal_1to1 \| smart)
  - `E05-0035`: template_id (design+writing)
  - `E05-0036`: add_sections (toc/executive_summary/findings/recommendations/appendix)
  - `E05-0037`: citation_mode (off/on + style)
  - `E05-0040`: ثم تشغيل واحد.
- **الحالة الحالية**: غير منفّذ
- **خطوات التنفيذ**:
  1. بناء خط أنابيب (pipeline) لوضع `AUTO` الذي يأخذ مدخلاً بسيطاً وينفذ سلسلة الخطوات المحددة في `E05-0028` بشكل مؤتمت بالكامل.
  2. تطوير واجهة الإعدادات في `Side Panel` لوضع `CONTROLLED` للسماح للمستخدم بضبط كافة المعلمات (knobs) المذكورة من `E05-0031` إلى `E05-0039`.
  3. إنشاء مشغل تنفيذ موحد (Unified Executor) يستدعي خط الأنابيب المناسب مع المعلمات المحددة بناءً على الوضع المختار.

#### المهمة 3.2: تنفيذ أنماط دقة المحتوى (LITERAL و SMART)
- **المتطلبات المشمولة**:
  - `E05-0041`: MUST تضمين نص المستخدم حرفيًا 1:1.
  - `E05-0042`: MUST الحفاظ على فواصل الأسطر/الفقرات.
  - `E05-0043`: التوزيع عبر صفحات يتم فقط عبر page breaks دون تغيير النص.
  - `E05-0044`: MUST إنتاج `literal_hash_in` و `literal_hash_out`.
  - `E05-0045`: MUST equality أو FAIL.
  - `E05-0046`: MUST تحليل الطلب + البيانات + الهدف + الجمهور.
  - `E05-0047`: MUST كتابة محتوى احترافي وفق tone + template writing style.
  - `E05-0048`: MUST عدم اختلاق حقائق (استخدام placeholder موسوم للبيانات غير المتوفرة).
  - `E05-0049`: MUST إنتاج “Content Trace” لربط كل ادعاء بمصدره.
- **الحالة الحالية**: غير منفّذ
- **خطوات التنفيذ**:
  1. تطوير معالج `MODE_LITERAL_1TO1` الذي يضمن إدراج النص كما هو تماماً، مع تنفيذ آلية التحقق عبر `literal_hash`.
  2. بناء معالج `MODE_SMART` الذي يستخدم نماذج اللغة لتحليل المدخلات وتوليد محتوى احترافي يتوافق مع القوالب والنبرة المحددة.
  3. تطبيق آلية `Content Trace` في وضع `SMART` لتسجيل مصدر كل معلومة أو رقم يتم إدراجه في التقرير.
  4. التأكد من أن معالج `SMART` يضع علامات واضحة (placeholders) للبيانات المطلوبة وغير الموجودة بدلاً من اختلاقها.

### المرحلة 4: إدارة القوالب والأنماط (Templates & Styling)

#### المهمة 4.1: تعريف وإدارة أنواع القوالب
- **المتطلبات المشمولة**:
  - `E05-0050`: **Brand Kit**: ألوان/خطوط/شعار/هوامش
  - `E05-0051`: **Report Template**: Layout + Styles + Numbering + Headers/Footers + TOC style
  - `E05-0052`: **Writing Template**: نبرة/أسلوب/مصطلحات/صيغ مخاطبة/افتتاح/ختام/صياغة توصيات
- **الحالة الحالية**: غير منفّذ
- **خطوات التنفيذ**:
  1. إنشاء نموذج بيانات لـ `Brand Kit` لتخزين الألوان، الخطوط، الشعارات، والهوامش.
  2. تطوير نموذج بيانات لـ `Report Template` لتعريف التخطيط، الأنماط، الترقيم، الرؤوس/التذييلات، ونمط جدول المحتويات.
  3. تطبيق نموذج بيانات لـ `Writing Template` لالتقاط قواعد النبرة، الأسلوب، المصطلحات، والصياغة.
  4. بناء واجهة لإدارة القوالب ضمن `Side Panel` لإنشاء وتعديل وتطبيق هذه القوالب.

#### المهمة 4.2: استخلاص القوالب من مستندات DOCX
- **المتطلبات المشمولة**:
  - `E05-0053`: من DOCX template MUST استخراج:
  - `E05-0054`: Styles (Heading 1..n, Body, Caption, Table styles)
  - `E05-0055`: Numbering & bullets schemes
  - `E05-0056`: Page setup (margins, paper size, RTL)
  - `E05-0057`: Headers/footers (with rules)
  - `E05-0058`: TOC configuration
  - `E05-0059`: Cover page structure
  - `E05-0060`: Writing ruleset (greeting patterns, tone lexicon, etc.)
- **الحالة الحالية**: غير منفّذ
- **خطوات التنفيذ**:
  1. تنفيذ محلل DOCX لاستخلاص الأنماط، خطط الترقيم، وإعدادات الصفحة.
  2. تطوير منطق لتحليل وتخزين محتوى وقواعد الرؤوس والتذييلات.
  3. إضافة وظيفة لاستخلاص إعدادات جدول المحتويات (TOC).
  4. بناء محرك قواعد لاستنتاج واستخلاص قواعد الكتابة (`Writing Ruleset`) من محتوى المستند.

#### المهمة 4.3: تطبيق قفل القالب (Template-Lock)
- **المتطلبات المشمولة**:
  - `E05-0061`: إذا user فعّل template-lock:
  - `E05-0062`: MUST الالتزام بالـstyles والـwriting rules حرفيًا.
  - `E05-0063`: MUST عدم إدخال Style جديد غير موجود.
  - `E05-0064`: MUST Report Compliance Report داخل Evidence Pack.
- **الحالة الحالية**: غير منفّذ
- **خطوات التنفيذ**:
  1. إضافة خيار تفعيل `template-lock` في إعدادات التقرير.
  2. عند التفعيل، يجب على محرك توليد التقرير الالتزام الصارم بأنماط وقواعد الكتابة الخاصة بالقالب المطبق.
  3. تنفيذ عمليات تحقق لمنع إدخال أي أنماط جديدة غير معرّفة.
  4. توليد `Template Compliance Report` كجزء من `Evidence Pack` للتحقق من الالتزام.

#### المهمة 4.4: تخصيص المحتوى للجمهور والمخاطب
- **المتطلبات المشمولة**:
  - `E05-0065`: المحرك MUST يدعم: recipient_title, organization_name, department, etc.
  - `E05-0066`: recipient_title (معالي/سعادة/الأستاذ/الأستاذة/CEO…)
  - `E05-0067`: recipient_name (اختياري)
  - `E05-0068`: organization_name
  - `E05-0069`: department
  - `E05-0070`: tone_profile
  - `E05-0071`: formality_level
  - `E05-0072`: MUST اختيار افتتاح/ختام وفق: اللغة، النبرة، قالب الكتابة، وسياق المؤسسة.
  - `E05-0073`: MUST عدم استخدام عبارات غير مناسبة ثقافيًا في العربية.
  - `E05-0074`: MUST حفظ “writing signature blocks” كقوالب قابلة لإعادة الاستخدام.
- **الحالة الحالية**: غير منفّذ
- **خطوات التنفيذ**:
  1. إنشاء نموذج بيانات لملف تعريف `Audience & Recipient`.
  2. تنفيذ محرك قواعد لاختيار التحيات والعبارات الختامية المناسبة بناءً على اللغة والنبرة والسياق التنظيمي.
  3. تطوير ميزة لحفظ وإعادة استخدام 
مكعبات توقيع الكتابة (writing signature blocks).

### المرحلة 5: التكامل مع البيانات (Data Integration)

#### المهمة 5.1: تطوير منتقي البيانات ووصفات التحويل (Data Picker & T-IR)
- **المتطلبات المشمولة**:
  - `E05-0093`: المستخدم MUST يستطيع اختيار dataset/table/columns.
  - `E05-0094`: تحديد filters/joins/transforms (T-IR).
  - `E05-0095`: معاينة البيانات الناتجة.
  - `E05-0096`: إدارة مصادر البيانات (إضافة/تحديث).
  - `E05-0097`: ربط كل مصدر بيانات بـ `dataset_signature`.
- **الحالة الحالية**: غير منفّذ
- **خطوات التنفيذ**:
  1. تطوير واجهة `Data Picker` ضمن `Side Panel` لعرض مصادر البيانات المتاحة.
  2. تصميم وتنفيذ نموذج التمثيل الوسيط للتحويلات (T-IR) لتعريف عمليات الفلترة والربط.
  3. بناء واجهة مرئية لإنشاء وتعديل وصفات T-IR.
  4. إضافة وظيفة لمعاينة البيانات بعد تطبيق وصفة T-IR.
  5. تنفيذ آلية لحساب وتخزين `dataset_signature` لكل مصدر بيانات مستخدم.

#### المهمة 5.2: ربط المكعبات بالبيانات (Data-Bound Blocks)
- **المتطلبات المشمولة**:
  - `E05-0098`: Table block: structured table, column selection, conditional formatting.
  - `E05-0099`: Chart block: embed chart with data, reflect filtered dataset.
  - `E05-0100`: KPI cards: measure + threshold + trend.
  - `E05-0014`: **Data-Bound**: الجداول/المخططات/الأرقام MUST تكون مربوطة بمصادر بيانات فعلية أو Placeholder موسوم.
- **الحالة الحالية**: غير منفّذ
- **خطوات التنفيذ**:
  1. تعديل مكعب `Table` في DOC-IR لربطه بمرجع بيانات (`data_binding_ref`).
  2. تطوير منطق يقوم بجلب البيانات وتعبئة جدول DOCX منظم، مع دعم اختيار الأعمدة وتنسيقها الشرطي.
  3. تطوير مكعب `Chart` لإنشاء مخططات (مثل الأعمدة، الخطوط، الدوائر) وتضمينها ككائنات قابلة للتحرير مع بياناتها.
  4. إنشاء مكعب `KPI Cards` لعرض مقاييس فردية مع عتبات واتجاهات.

#### المهمة 5.3: آلية تحديث البيانات (Data Refresh)
- **المتطلبات المشمولة**:
  - `E05-0101`: عند تحديث البيانات/الوصفات: MUST تحديث التقرير، إنشاء نسخة جديدة، وتسجيل lineage.
- **الحالة الحالية**: غير منفّذ
- **خطوات التنفيذ**:
  1. إضافة زر "تحديث البيانات" في الواجهة.
  2. عند الضغط، يتم إعادة تنفيذ جميع استعلامات T-IR المرتبطة بالتقرير.
  3. تحديث محتوى جميع المكعبات المرتبطة بالبيانات.
  4. إنشاء نسخة جديدة من التقرير تلقائياً (Versioning).
  5. تسجيل عملية التحديث في سجل التدقيق (Audit Log) مع تفاصيل المصادر والنسخ.

### المرحلة 6: الحوكمة والامتثال (Governance & Compliance)

#### المهمة 6.1: تطبيق تصنيف المستندات وسياسات المشاركة
- **المتطلبات المشمولة**:
  - `E05-0105`: each report MUST have classification: public/internal/confidential/restricted.
  - `E05-0106`: MUST enforce export/share policy by classification.
  - `E05-0017`: **Classification + Governance**: تصنيف الوثيقة + سياسة مشاركة + صلاحيات + سجل تدقيق + نسخ + اعتماد.
  - `E05-0038`: sensitivity_classification (public/internal/confidential/restricted)
- **الحالة الحالية**: غير منفّذ
- **خطوات التنفيذ**:
