_# الخطة التنفيذية الشاملة — محرك راصد الذكي (E08)

## ملخص تنفيذي
محرك راصد الذكي (E08) هو نظام التشغيل المركزي لمنصة راصد، مصمم ليكون العقل المدبر الذي يدير كافة العمليات والمهام. يتجاوز المحرك مفهوم "الشات بوت" التقليدي ليقدم نظامًا وكيلاً (Agentic System) متكاملًا يعمل ضمن واجهة موحدة (Canvas). يقوم المحرك بفهم أهداف المستخدم، ووضع خطط تنفيذية دقيقة، وتشغيل كافة محركات المنصة الأخرى (مثل محركات التحويل، المطابقة، العروض، التقارير) بشكل آلي وموثوق. من أبرز وظائفه الأساسية، القدرة على التحكم الكامل بالمنصة، وتدريب نفسه وتطوير مهاراته عبر "مركز تدريب" متخصص، وتقديم جولات إرشادية حية (Guided Tours) للمستخدمين داخل الواجهة. يرتكز تصميم المحرك على مبادئ الحتمية (Determinism)، والموثوقية (Truthfulness)، ومنع أي تنفيذ وهمي أو ادعاء غير مدعوم بدليل، مما يضمن أن كل إجراء يتم تسجيله والتحقق منه بشكل صارم.

## ثالثاً: الخطة التنفيذية المرحلية

### المرحلة 1: تأسيس المعمارية الأساسية ونواة نظام التشغيل

#### المهمة 1.1: بناء المكونات الهيكلية للمحرك
- **المتطلبات المشمولة**:
  - `E08-0031`: راصد MUST يبنى كنظام متعدد الطبقات، وليس “prompt واحد”.
  - `E08-0032`: **Intent Engine** ↳ prompt+assets → `intent_manifest` كامل
  - `E08-0033`: **Policy Engine** ↳ permissions + classifications + strict claims + risk gates
  - `E08-0034`: **Planner (Action Graph Builder)** ↳ intent_manifest → `action_graph_plan` (deterministic)
  - `E08-0035`: **Executor (Action Runtime Client)** ↳ يشغل الأدوات (tools) ويُتابع progress ويعالج retries
  - `E08-0036`: **Verifier (Gates Orchestrator)** ↳ pixel gates / structural gates / lqa gates / transcript gates / parity gates
  - `E08-0037`: **Evidence Producer** ↳ يجمع الأدلة ويُرجع `evidence_id` قبل أي “Done”
  - `E08-0044`: **Telemetry & Audit** ↳ logs, metrics, traces, immutable audit
- **الحالة الحالية**: غير منفّذ
- **خطوات التنفيذ**:
  1. تصميم وتنفيذ البنية متعددة الطبقات للمحرك.
  2. تطوير محرك النوايا (Intent Engine) لتحليل مدخلات المستخدم وتحويلها إلى `intent_manifest`.
  3. بناء محرك السياسات (Policy Engine) لفرض الصلاحيات وقواعد الحوكمة.
  4. إنشاء المخطط (Planner) لتوليد خطط تنفيذ حتمية (`action_graph_plan`).
  5. تطوير المنفذ (Executor) للتفاعل مع `Action Runtime` وتشغيل الأدوات.
  6. بناء نظام التحقق (Verifier) لتنسيق بوابات التحقق من جودة المخرجات.
  7. تطوير منتج الأدلة (Evidence Producer) لربط المخرجات بأدلة ملموسة.
  8. تأسيس نظام التسجيل والمراقبة (Telemetry & Audit).

#### المهمة 1.2: ضمان الحتمية والموثوقية
- **المتطلبات المشمولة**:
  - `E08-0045`: أي خطة تنفيذ MUST تكون deterministic: ↳ نفس المدخلات + نفس السياسات + نفس versions => نفس action_graph
  - `E08-0046`: أي مخرجات MUST تكون reproducible: ↳ action_snapshot + tool_versions + inputs hashes.
  - `E08-0004`: **No-Cheating**: راصد MUST NOT يدّعي تنفيذًا لم يحدث. MUST لا يقول “تم” إلا بعد Evidence/Gates من Action Runtime.
  - `E08-0011`: **Truthfulness Contract** (منع الهلوسة): ↳ MUST كل رسالة “نفذت” تتضمن references إلى `action_ids` + `artifact_ids` + `evidence_id`.
- **الحالة الحالية**: غير منفّذ
- **خطوات التنفيذ**:
  1. تطبيق آليات لضمان أن نفس المدخلات تنتج نفس خطة العمل دائماً.
  2. تطوير نظام `snapshots` لتسجيل حالة الإجراءات والمدخلات لضمان قابلية استنساخ النتائج.
  3. ربط كل مخرجات المحرك بأدلة تنفيذية من `Action Runtime`.
  4. فرض عقد الموثوقية بحيث لا يتم إعلان أي نتيجة كـ "منفذة" بدون `evidence_id`.

### المرحلة 2: بناء واجهات التفاعل والتحكم

#### المهمة 2.1: تطوير آليات التواصل الأساسية
- **المتطلبات المشمولة**:
  - `E08-0016`: اللغة الافتراضية: العربية.
  - `E08-0017`: MUST يغيّر النبرة حسب إعداد المستخدم: ↳ رسمي حكومي / تجاري / تقني / مبسط
  - `E08-0018`: MUST يدعم أسلوب “مختصر جدًا” أو “تفصيلي” حسب preference.
  - `E08-0019`: MUST لا يطيل الشرح في الرسالة الأساسية.
  - `E08-0020`: MUST يستخدم: ↳ PlanCard مختصر ↳ Progress RunCard ↳ ثم يضع “تفاصيل” داخل collapsible card أو “Explain” عند الطلب.
  - `E08-0021`: MUST لا يسأل “هل تريد…؟” كسؤال حواري متكرر.
  - `E08-0022`: أي اختيار MUST يكون: ↳ Default policy أو ↳ Control chips داخل ContextActionsCard أو ↳ Command palette.
- **الحالة الحالية**: غير منفّذ
- **خطوات التنفيذ**:
  1. ضبط اللغة العربية كلغة افتراضية للمحرك.
  2. بناء نظام لتغيير نبرة وأسلوب الخطاب بناءً على إعدادات المستخدم.
  3. تصميم وتنفيذ بطاقات العرض (Cards) الموجزة للتخطيط والتقدم.
  4. تطبيق قاعدة "عدم السؤال" واستبدالها بعناصر تحكم واضحة وسياسات افتراضية.

#### المهمة 2.2: بناء نظام التحكم بواجهة المستخدم (UI Control)
- **المتطلبات المشمولة**:
  - `E08-0055`: كل عنصر تفاعلي في الواجهة MUST يحمل: `data-rased-id`.
  - `E08-0056`: `data-rased-id=\"<semantic_id>\"`
  - `E08-0057`: ممنوع الاعتماد على: CSS selectors, DOM order, النصوص المتغيرة.
  - `E08-0058`: CSS selectors
  - `E08-0059`: النصوص المتغيرة
  - `E08-0060`: راصد MUST يتحكم فقط عبر “UI Action API” داخل التطبيق.
  - `E08-0075`: التطبيق MUST يوفر snapshot API.
  - `E08-0083`: راصد MUST يستخدم snapshot قبل كل خطوة لتفادي الأخطاء.
- **الحالة الحالية**: غير منفّذ
- **خطوات التنفيذ**:
  1. فرض استخدام `data-rased-id` كمعرف أساسي ومستقر لجميع عناصر الواجهة التفاعلية.
  2. تطوير واجهة برمجة تطبيقات (API) مخصصة لإجراءات الواجهة (`UI Action API`) تكون هي الوسيلة الوحيدة لتحكم راصد بالواجهة.
  3. بناء `snapshot API` التي توفر لراصد لمحة عن حالة الواجهة الحالية (العناصر المحددة، الألواح المفتوحة، إلخ).
  4. دمج استدعاء `snapshot API` في دورة حياة المحرك قبل تنفيذ أي خطوة.

### المرحلة 3: تفعيل أنماط التشغيل والجولات الإرشادية

#### المهمة 3.1: تنفيذ أنماط التشغيل المختلفة
- **المتطلبات المشمولة**:
  - `E08-0023`: **AUTO Mode**: راصد يستنتج intent ويبدأ تنفيذ خطة كاملة دون أسئلة.
  - `E08-0024`: يعرض للمستخدم: ↳ PlanCard (3–6 خطوات) ↳ Controls مختصرة للتعديل (5–9) ↳ ثم تشغيل.
  - `E08-0025`: **CONTROLLED Mode**: المستخدم يحدد knobs قبل التنفيذ (بدون حوار).
  - `E08-0026`: ثم راصد ينفذ مرة واحدة.
  - `E08-0027`: **TUTOR Mode**: راصد لا يكتفي بالنتيجة؛ بل: ↳ يفتح Guided Tour ↳ يشرح أين يضغط المستخدم ↳ يضع أمثلة حيّة داخل canvas
  - `E08-0028`: TUTOR Mode MUST لا يفرض نفسه إلا إذا user طلب “علمني/أرشدني” أو إذا policy المؤسسة فعلت “Assistive Guidance”.
  - `E08-0029`: **EXECUTOR Mode**: راصد ينفذ “بالنيابة” إجراءات UI داخل canvas (فتح/تحديد/تطبيق) ضمن صلاحيات.
  - `E08-0030`: MUST يسجل كل خطوة في Action Runtime + Audit.
- **الحالة الحالية**: غير منفّذ
- **خطوات التنفيذ**:
  1. تطوير منطق `AUTO Mode` لتحليل النية وتوليد خطة تلقائية.
  2. بناء واجهة `CONTROLLED Mode` التي تسمح للمستخدم بضبط الإعدادات المسبقة.
  3. ربط `TUTOR Mode` بمحرك الجولات الإرشادية.
  4. تطوير `EXECUTOR Mode` لتنفيذ إجراءات الواجهة بالنيابة عن المستخدم مع تسجيلها بالكامل.

#### المهمة 3.2: بناء محرك الجولات الإرشادية الحية (Guided Tours)
- **المتطلبات المشمولة**:
  - `E08-0013`: **Guided Tours حيّة**: ↳ MUST راصد يستطيع أن “يذهب” داخل الواجهة.
  - `E08-0041`: **Guided Tour Engine** ↳ spotlight, callouts, ghost cursor, stepper
  - `E08-0084`: **Explain Mode**: يشرح فقط ويُظهر spotlight
  - `E08-0085`: **Coach Mode**: يشرح + يطلب من المستخدم تنفيذ click (step gating)
  - `E08-0086`: **Do It For Me Mode**: ينفذ UI actions بالنيابة + يشرح
  - `E08-0087`: Spotlight على عنصر (data-rased-id)
  - `E08-0088`: Callout bubble (نص قصير + زر Next)
  - `E08-0090`: Stepper progress (1/7)
  - `E08-0091`: “Try it” gating: ↳ في Coach: لا ينتقل حتى يقوم المستخدم بالفعل المطلوب
  - `E08-0092`: Auto-play: ↳ في Executor: راصد ينفذ الفعل ثم ينتقل
  - `E08-0093`: لا يغطي tour أكثر من 25% من الشاشة
  - `E08-0095`: يمكن إيقافه في أي لحظة
  - `E08-0097`: كل tour MUST يُسجل.
- **الحالة الحالية**: غير منفّذ
- **خطوات التنفيذ**:
  1. بناء المكونات الأساسية لمحرك الجولات الإرشادية (Spotlight, Callout, Stepper).
  2. تطوير الأنماط الثلاثة للإرشاد (Explain, Coach, Do It For Me).
  3. تطبيق آلية الربط بالفعل (`Try it` gating) لضمان تفاعل المستخدم في نمط `Coach`.
  4. تطوير نظام تسجيل وتحليل بيانات الجولات الإرشادية (الخطوات، معدل الإكمال، إلخ).

### المرحلة 4: إنشاء مركز التدريب ونظام المعرفة

#### المهمة 4.1: بناء مكونات مركز التدريب
- **المتطلبات المشمولة**:
  - `E08-0012`: **Training Center**: ↳ MUST وجود مركز تدريب كامل لراصد داخل المنصة.
  - `E08-0040`: **Training Center** ↳ packs/playbooks/skills/evaluations
  - `E08-0103`: **Knowledge Packs** ↳ مستندات المؤسسة, سياسات, مصطلحات, قوالب.
  - `E08-0104`: **Skill Packs** ↳ مهارة لكل محرك.
  - `E08-0105`: **Playbooks** ↳ سيناريوهات تشغيل جاهزة.
  - `E08-0106`: **Evaluation Harness** ↳ Golden corpora, Scenario replay, Pass/Fail rubrics.
  - `E08-0108`: **Certification** ↳ لا يُسمح لراصد بتنفيذ “Features حساسة” إلا بعد اجتياز evaluation.
- **الحالة الحالية**: غير منفّذ
- **خطوات التنفيذ**:
  1. تصميم وتنفيذ الواجهات لإدارة حزم المعرفة (Knowledge Packs) وحزم المهارات (Skill Packs).
  2. بناء نظام لإدارة وتخزين سيناريوهات التشغيل (Playbooks).
  3. تطوير أداة التقييم (Evaluation Harness) مع القدرة على إعادة تشغيل السيناريوهات وتقييم النتائج.
  4. تطبيق آلية "الشهادة" التي تربط تفعيل الميزات الحساسة باجتياز التقييمات.

#### المهمة 4.2: تطوير نظام إدارة المعرفة والتخصيص
- **المتطلبات المشمولة**:
  - `E08-0038`: **Memory & Personalization** ↳ user_prefs / org_prefs / templates / term packs / saved recipes
  - `E08-0039`: **Knowledge System (RAG داخلي)** ↳ workspace scoped retrieval فقط
  - `E08-0109`: knowledge ingestion MUST يكون: ↳ tenant-scoped, versioned, auditable
  - `E08-0111`: MUST دعم termbase: ↳ ar/en, domains, preferred/forbidden terms
  - `E08-0112`: MUST دعم style guide: ↳ مخاطبة الجهات, جُمل افتتاح/ختام, لهجة حكومية/تجارية
- **الحالة الحالية**: غير منفّذ
- **خطوات التنفيذ**:
  1. بناء نظام الذاكرة والتخصيص لتخزين تفضيلات المستخدم والمؤسسة.
  2. تطوير نظام معرفة داخلي (RAG) يقتصر على استرجاع المعلومات من نطاق `workspace` فقط.
  3. إنشاء عملية استيعاب معرفة (ingestion) تضمن العزل، وتحديد الإصدارات، وقابلية التدقيق.
  4. تطوير ودعم قاعدة المصطلحات (Termbase) وأدلة الأسلوب (Style Guides) للتحكم بلغة ونبرة المحرك.

### المرحلة 5: تنفيذ Playbooks والربط الخارجي

#### المهمة 5.1: بناء وتشغيل سيناريوهات التشغيل (Playbooks)
- **المتطلبات المشمولة**:
  - `E08-0113`: playbook هو ملف قابل للتنفيذ.
  - `E08-0114`: trigger: intent patterns
  - `E08-0115`: steps: action_graph template
  - `E08-0116`: validations: gates
  - `E08-0117`: outputs: expected artifacts
  - `E08-0119`: PDF → PPTX strict
  - `E08-0120`: Image table → XLSX strict
  - `E08-0121`: 50 xlsx → unify → dashboard → report → slides
  - `E08-0122`: Video → transcript strict → localize → report
  - `E08-0123`: Dataset → executive dashboard + share + exports
  - `E08-0124`: MUST playbook يحدد كل defaults
- **الحالة الحالية**: غير منفّذ
- **خطوات التنفيذ**:
  1. تعريف بنية ملف الـ `playbook` القابل للتنفيذ.
  2. تطوير آلية لتشغيل الـ `playbooks` بناءً على أنماط النوايا (intent patterns).
  3. تنفيذ قوالب `playbook` محددة مسبقًا للسيناريوهات المعقدة (مثل `PDF → PPTX`).
  4. ضمان أن جميع الـ `playbooks` تحدد قيمها الافتراضية لتجنب الأسئلة.

#### المهمة 5.2: تطوير مدير الربط الآمن (Connector Manager)
- **المتطلبات المشمولة**:
  - `E08-0043`: **Connector Manager (APIs)** ↳ call external APIs عبر connectors آمنة + vault + allowlist
  - `E08-0126`: جميع الاستدعاءات الخارجية MUST تمر عبر: ↳ connector registry, secrets vault, domain allowlist, rate limits, audit logging
  - `E08-0127`: MUST عدم إرسال بيانات حساسة خارج المؤسسة إذا policy تمنع
  - `E08-0128`: MUST احترام classification: ↳ restricted → ممنوع خروج, confidential → allow only approved connectors
- **الحالة الحالية**: غير منفّذ
- **خطوات التنفيذ**:
  1. بناء مدير الربط (Connector Manager) كبوابة إلزامية لجميع الاتصالات الخارجية.
  2. دمج `secrets vault` لإدارة بيانات الاعتماد بشكل آمن.
  3. تطبيق قوائم السماح (allowlists) للتحكم بالوجهات المسموح بها.
  4. فرض سياسات أمان البيانات بناءً على تصنيفها لمنع تسرب المعلومات الحساسة.

### المرحلة 6: تطبيق بوابات الأمان والتدقيق والأدوات النهائية

#### المهمة 6.1: فرض بوابات الأمان والتدقيق
- **المتطلبات المشمولة**:
  - `E08-0008`: **أمان وحقوق**: راصد لا ينفّذ عمليات خطرة إلا عبر “Explicit Command Gate”.
  - `E08-0129`: delete dataset
  - `E08-0130`: publish public link
  - `E08-0131`: revoke permissions
  - `E08-0132`: overwrite templates
  - `E08-0134`: explicit command token داخل الشات مثل: ↳ `CONFIRM PUBLISH`
  - `E08-0136`: أي فعل خطر MUST يسجل: ↳ user_id, timestamp, action_id
  - `E08-0137`: before/after snapshot
- **الحالة الحالية**: غير منفّذ
- **خطوات التنفيذ**:
  1. تطوير آلية "بوابة الأوامر الصريحة" (Explicit Command Gate) للأفعال الخطرة.
  2. فرض استخدام `tokens` تأكيدية (مثل `CONFIRM DELETE`) كشرط للتنفيذ.
  3. بناء نظام تدقيق يسجل تفاصيل كل فعل خطر مع `snapshots` للحالة قبل وبعد التنفيذ.

#### المهمة 6.2: تأسيس سجل الأدوات والعقود النهائية
- **المتطلبات المشمولة**:
  - `E08-0141`: راصد لا ينفذ شيئًا “داخل عقله”. ينفذ عبر Tools فقط.
  - `E08-0142` - `E08-0159`: (قائمة الأدوات الحرجة مثل `rased.intent_parse`, `rased.plan_action_graph`, إلخ).
  - `E08-0160`: أي Tool بدون schema/permissions/determinism metadata = غير قابل للتشغيل.
  - `E08-0052`: MUST NOT كتابة “تم/اكتمل/جاهز” إلا إذا: `action_graph.status == completed` و `evidence_id` موجود.
  - `E08-0053`: MUST كل claim “أنشأت/حوّلت/صدّرت” يرفق: `action_ids[]`, `artifact_ids[]`, `evidence_id`.
  - `E08-0054`: MUST عند الفشل: ↳ يعرض “Repair Plan” + “Retry” ولا يختلق نتيجة.
- **الحالة الحالية**: غير منفّذ
- **خطوات التنفيذ**:
  1. بناء سجل الأدوات (Tool Registry) وتعبئته بمجموعة الأدوات الحرجة المطلوبة.
  2. فرض وجود `schema` وبيانات وصفية إلزامية لكل أداة مسجلة.
  3. تطبيق القواعد الصارمة النهائية لرسائل الإكمال، بحيث تكون مرتبطة بحالة `action_graph` والأدلة.
  4. تطوير آلية "خطة الإصلاح" (Repair Plan) للتعامل مع حالات الفشل بشكل شفاف.
