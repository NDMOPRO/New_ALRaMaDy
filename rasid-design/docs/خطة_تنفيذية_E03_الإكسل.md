> # الخطة التنفيذية الشاملة — محرك الإكسل (E03)

## ملخص تنفيذي

تهدف هذه الوثيقة إلى تقديم خطة تنفيذية فنية شاملة ومكتفية ذاتياً لتطوير **محرك الإكسل (E03)**. تم تصميم هذا المحرك ليكون أداة فائقة القدرة على نطاق واسع، تتيح للمستخدمين استيعاب، فهم، تنظيف، توحيد، دمج، تقسيم، تحويل، تحليل، وتصدير أي نوع من البيانات، سواء كانت من ملفات Excel, CSV, PDF, أو حتى صور. 

يعتمد المحرك على واجهة مستخدم مركزية تُعرف بـ **"Data Canvas"**، والتي تبدأ بجدول فارغ. يمكن للمستخدم سحب وإفلات أعمدة من مصادر بيانات متعددة ومختلفة (ملفات، أوراق عمل، جداول) لبناء جدوله النهائي، مع دعم كامل للتعامل مع مجموعات بيانات ضخمة تصل إلى ملايين أو مليارات السجلات بفضل البنية التحتية القائمة على **Columnar Lakehouse** وتقنيات العرض الافتراضي (Virtualization). 

تتضمن القدرات الأساسية للمحرك ما يلي:
- **محرك تحويلات مرئي (Visual Transformation Engine)**: واجهة مرئية شبيهة بـ Power Query تتيح بناء عمليات معالجة البيانات المعقدة بدون الحاجة لكتابة أي كود، مع القدرة على تصدير هذه التحويلات إلى لغة M الخاصة بـ Power Query عند الطلب.
- **آلة إكسل افتراضية (Spreadsheet Virtual Machine - SVM)**: محرك داخلي يضمن تنفيذ الصيغ والمعادلات بنفس الدقة والموثوقية الموجودة في Excel، مع دعم للجداول المحورية، التنسيق الشرطي، وتثبيت الأجزاء.
- **محلل بيانات ذكي (AI Analyst)**: قدرات ذكاء اصطناعي متقدمة لا تقتصر على المساعدة، بل تقوم بتحليل سياق البيانات بشكل استباقي لاقتراح عمليات تنظيف، توحيد، دمج، تحديد مؤشرات أداء رئيسية (KPIs)، والمساعدة في تحليل الأسباب الجذرية والتنبؤ.

تتبع الخطة منهجية صارمة تمنع أي شكل من أشكال الادعاءات الوهمية أو الحلول المؤقتة، حيث يُعتبر أي سلوك غير منصوص عليه في المتطلبات ممنوعاً تماماً. كل متطلب سيتم تنفيذه واختباره بشكل كامل قبل اعتباره "منجزاً".

## ثالثاً: الخطة التنفيذية المرحلية

### المرحلة 1: الأساس والبنية التحتية (Foundation & Infrastructure)

#### المهمة 1.1: إعداد بيئة التطوير وتأسيس واجهة المستخدم الرئيسية (Data Canvas)
- **المتطلبات المشمولة**:
  - `E03-0001`: **وثيقة تنفيذية فنية موجهة للمنفّذ مباشرة** — صياغة أمرية إلزامية (MUST/SHALL/MUST NOT)
  - `E03-0003`: **قاعدة صارمة**: أي سلوك غير منصوص عليه هنا = **ممنوع**. أي “تقريب/ديمو/ادعاء” = **مرفوض**. لا يُقال “تم” إلا بعد تنفيذ حقيقي + اختبارات + Evidence.
  - `E03-0010`: **No-Cheating / No Dummy / No Claims**
  - `E03-0011`: **شاشة واحدة فقط: Data Canvas**
    - وسط الشاشة: **Result Table** (جدول النتائج) + Tabs للأوراق داخل نفس الملف الناتج.
    - يسار/يمين: Panel واحد قابل للإخفاء يحتوي على: Library, Column Map, Operations, Recipes, QA/Quality, Export.
  - `E03-0012`: **MUST عدم عرض خيارات كثيرة دفعة واحدة**.
  - `E03-0013`: **MUST عرض 5–9 خيارات فقط حسب السياق**.
  - `E03-0014`: **MUST توفير “Search Controls” للعثور على أي عملية بالاسم**.
- **الحالة الحالية**: غير منفّذ
- **خطوات التنفيذ**:
  1. إعداد هيكل المشروع الأساسي للتطبيق (Frontend/Backend).
  2. تصميم وتنفيذ المكون الرئيسي لواجهة المستخدم `DataCanvas`.
  3. بناء اللوحة الجانبية القابلة للإخفاء مع التبويبات المبدئية (Library, Operations, etc.).
  4. تطوير آلية التحكم التدريجي (Progressive Controls) التي تعرض الأوامر بناءً على سياق تحديد المستخدم (عمود، جدول، لا شيء).
  5. إضافة حقل البحث عن الأوامر (`Search Controls`).
  6. التأكد من أن جميع المكونات الوهمية (placeholders) تحمل علامة واضحة `DUMMY_COMPONENT` لمنع اعتبارها منفذة.

#### المهمة 1.2: بناء طبقة التخزين الأساسية (Canonical Storage) ومحرك التنفيذ
- **المتطلبات المشمولة**:
  - `E03-0035`: **MUST تحويل كل البيانات داخليًا إلى Columnar**: Arrow in-memory + Parquet on-disk.
  - `E03-0036`: **MUST وجود Catalog**: file → sheet → table → column lineage, column fingerprints + stats + semantic tags.
  - `E03-0037`: **MUST وجود Semantic Graph**: entities, keys, relationships, time dimensions.
  - `E03-0038`: **كل كيان MUST له معرف**.
  - `E03-0039`: **asset_id (file)**, sheet_id, table_id, column_id.
  - `E03-0040`: **dataset_id (نتيجة)**.
  - `E03-0041`: **recipe_id (سير عمل)**.
  - `E03-0098`: **MUST تنفيذ استعلامات وتحويلات عبر**: Arrow + embedded analytical engine (DuckDB-class), with optional distributed engine (ClickHouse-class) at scale.
  - `E03-0100`: **result table must be virtualized**: no full load, fetch windowed rows.
  - `E03-0101`: **all operations run async with progress**.
- **الحالة الحالية**: غير منفّذ
- **خطوات التنفيذ**:
  1. دمج مكتبة Apache Arrow للتعامل مع البيانات في الذاكرة.
  2. إعداد آلية لتخزين البيانات على القرص باستخدام صيغة Parquet.
  3. تصميم وتنفيذ مخطط قاعدة البيانات أو بنية الملفات الخاصة بـ **Catalog** لتتبع أصل البيانات (lineage) والبيانات الوصفية (metadata).
  4. بناء نظام لإنشاء وتعيين المعرفات الفريدة (IDs) لجميع الكيانات (Assets, Datasets, Recipes, etc.).
  5. دمج محرك استعلامات تحليلي مثل DuckDB لتنفيذ العمليات على بيانات Arrow.
  6. تطوير الواجهة البرمجية (API) الخاصة بجدول النتائج لدعم العرض الافتراضي (virtualized rendering) وجلب البيانات على شكل نوافذ (windowed rows).
  7. تأسيس نظام إدارة المهام في الخلفية (Job Queue) لضمان تنفيذ جميع العمليات الثقيلة بشكل غير متزامن مع إظهار شريط تقدم.


### المرحلة 2: استيعاب البيانات والتحليل المبدئي (Core Data Ingestion & Initial Analysis)

#### المهمة 2.1: تطوير آلية استيعاب المصادر المتعددة (Multi-Source Ingestion)
- **المتطلبات المشمولة**:
  - `E03-0028`: **دعم استيراد ملفات Excel (xlsx/xlsm) و CSV/TXT**.
  - `E03-0033`: **دعم استيراد حزم ZIP ومجلدات كاملة**.
  - `E03-0042`: **MUST دعم سحب وإفلات الملفات المتعددة، المجلدات، وحزم ZIP دفعة واحدة**.
  - `E03-0029`: **دعم استيراد ملفات PDF (جداول/تقارير)**.
  - `E03-0030`: **دعم استيراد الصور التي تحتوي على جداول أو تقارير**.
  - `E03-0034`: **PDF/Images: MUST استخدام محرك “Strict Table Extraction”** لاستخراج الجداول بشكل منظم مع الحفاظ على الأنماط.
- **الحالة الحالية**: غير منفّذ
- **خطوات التنفيذ**:
  1. بناء الواجهة البرمجية (API) الخلفية لاستقبال الملفات والمجلدات.
  2. تطوير منطق التعامل مع حزم ZIP لفك ضغطها ومعالجة محتوياتها.
  3. دمج مكتبات متخصصة لقراءة ملفات Excel (xlsx/xlsm) و CSV، مع معالجة ترميزات مختلفة.
  4. دمج محرك "Strict Table Extraction" (من منظومة المطابقة) لمعالجة ملفات PDF والصور.
  5. تطوير الواجهة الأمامية (UI) في لوحة `Library` للسماح للمستخدم برفع الملفات والمجلدات عبر السحب والإفلات.
  6. التأكد من أن عملية الرفع تتم في الخلفية (background) مع إظهار مؤشر للتقدم.

#### المهمة 2.2: المسح المبدئي وبناء خريطة المحتوى (Preflight Scan & Content Map)
- **المتطلبات المشمولة**:
  - `E03-0043`: **يجب إجراء مسح مبدئي قبل إدخال البيانات إلى الـ lakehouse**.
  - `E03-0044`: **MUST عرض ملخص** يتضمن: عدد الشيتات/الجداول، عدد الصفوف/الأعمدة، نسبة الفراغات، الأعمدة الحساسة (PII) المكتشفة، الأعمدة المتكررة، واحتمالات الربط.
  - `E03-0045`: **MUST بناء “Content Map”**: خريطة مرئية أو هيكلية لكل ملف ← شيت ← جدول ← عمود.
- **الحالة الحالية**: غير منفّذ
- **خطوات التنفيذ**:
  1. بعد رفع الملفات وقبل تخزينها في Parquet، يتم تشغيل عملية مسح سريعة.
  2. تطوير وحدات تحليلية (analyzers) لكل نوع ملف لحساب الإحصائيات المطلوبة (عدد الصفوف، الفراغات، إلخ).
  3. دمج آلية كشف البيانات الشخصية (PII detection) لتحليل الأعمدة.
  4. تصميم وتخزين بنية بيانات `ContentMap` التي تمثل هيكل كل ملف مصدر.
  5. عرض ملخص المسح وخريطة المحتوى في واجهة المستخدم ليقوم المستخدم بمراجعتها قبل تأكيد عملية الاستيعاب الكاملة.

#### المهمة 2.3: تأسيس وظيفة سحب وإفلات الأعمدة عبر الملفات
- **المتطلبات المشمولة**:
  - `E03-0002`: **نطاق المحرك**: ... مع واجهة “جدول فارغ” + سحب وإفلات أعمدة من أي ملف/ورقة/جدول...
  - `E03-0005`: **Canvas واحد + جدول فارغ كمدخل**: MUST يبدأ المستخدم من “Blank Table Canvas” يمكنه أن يبنيه بالسحب والإفلات.
  - `E03-0006`: **Cross-File Column Drag & Drop**: MUST تمكين سحب عمود من أي ملف/أي ورقة/أي جدول وإفلاته داخل الجدول النهائي.
- **الحالة الحالية**: غير منفّذ
- **خطوات التنفيذ**:
  1. في لوحة `Library`، يتم عرض الأعمدة المتاحة من `ContentMap` لكل مصدر.
  2. تفعيل خاصية السحب (Draggable) على عناصر الأعمدة في المكتبة.
  3. جعل منطقة `Result Table` في الـ `DataCanvas` قابلة للإفلات (Droppable).
  4. عند إفلات عمود، يتم إرسال طلب إلى الواجهة الخلفية لبدء عملية جلب بيانات هذا العمود من مصدره (المخزن في Parquet) وإضافته إلى `Result Table`.
  5. تحديث `Result Table` (باستخدام العرض الافتراضي) لإظهار العمود الجديد وبياناته الأولية.


### المرحلة 3: محرك التحويلات ومعالجة البيانات (T-IR Engine)

#### المهمة 3.1: بناء محرك التحويلات الوسيط (T-IR) والعمليات الأساسية على الأعمدة
- **المتطلبات المشمولة**:

  - `E03-0063`: **Rename فردي/جماعي**.
  - `E03-0064`: **Enforce naming standard (snake_case)**.
  - `E03-0065`: **Keep original display name as alias**.
  - `E03-0066`: **Split by delimiter**.
  - `E03-0067`: **Split by regex (pro)**.
  - `E03-0068`: **Split date into y/m/d**.
  - `E03-0071`: **Merge with format template**.
  - `E03-0074`: **Trim spaces**.
  - `E03-0075`: **Normalize unicode**.
  - `E03-0076`: **Remove symbols, change casing**.
  - `E03-0079`: **Detect & convert types** (dates, currency, percent).
  - `E03-0080`: **MUST not corrupt IDs (leading zeros preserved)**.
  - `E03-0081`: **Create column from expression**.
- **الحالة الحالية**: غير منفّذ
- **خطوات التنفيذ**:
  1. تصميم وتوثيق مواصفة `Transform Intermediate Representation (T-IR)` بشكل كامل.
  2. تطوير واجهة مرئية (UI) في لوحة `Operations` لإضافة خطوات التحويل.
  3. تنفيذ العمليات الأساسية كخطوات T-IR: `rename`, `split`, `merge`, `clean`, `cast`.
  4. بناء محرك выражений (Expression Engine) لإنشاء أعمدة مشتقة.
  5. ضمان أن كل عملية تحويل يتم تسجيلها كخطوة منفصلة في `Recipe`.

#### المهمة 3.2: تطوير قدرات الربط والتجميع (Joins & Aggregations)
- **المتطلبات المشمولة**:
  - `E03-0054`: **MUST دعم أنواع الربط**: inner/left/right/full/semi/anti.
  - `E03-0055`: **MUST دعم composite keys**.
  - `E03-0057`: **MUST دمج جداول متشابهة (Union/Append)** عبر محاذاة المخطط وتوحيد الأعمدة.
  - `E03-0059`: **MUST دعم عمليات التجميع**: sum/avg/count/distinct_count/min/max/std.
  - `E03-0060`: **MUST دعم Pivot/Unpivot**.
  - `E03-0061`: **MUST إنشاء “Summary Tables” بضغطة زر**.
- **الحالة الحالية**: غير منفّذ
- **خطوات التنفيذ**:
  1. تطوير واجهة مرئية لتحديد جداول الربط، المفاتيح، ونوع الربط.
  2. تنفيذ منطق الربط (Join) في محرك الاستعلامات (DuckDB) بناءً على مواصفات T-IR.
  3. تنفيذ عملية الدمج (Union) مع المعالجة الذكية للأعمدة المفقودة.
  4. بناء واجهة مرئية لتحديد أعمدة التجميع (group by) والمقاييس (aggregations).
  5. تنفيذ عمليات التجميع والـ Pivot/Unpivot كخطوات T-IR.

#### المهمة 3.3: تطبيق نموذج المعاينة الفورية والتنفيذ غير المتزامن

  - `E03-0084`: **T-IR MUST يدعم preview سريع على sample**.
  - `E03-0085`: **T-IR MUST يتم تطبيقه بالكامل على lakehouse**.
- **الحالة الحالية**: غير منفّذ
- **خطوات التنفيذ**:
  1. تعديل محرك T-IR ليدعم وضعين: `Preview` (يعمل على عينة صغيرة وسريعة) و `Apply` (يعمل على كامل البيانات).
  2. عند إضافة أو تعديل أي خطوة T-IR في الواجهة، يتم استدعاء وضع `Preview` تلقائيًا.
  3. تحديث `Result Table` بناءً على نتيجة المعاينة فورًا.
  4. عند ضغط المستخدم على زر "Apply"، يتم إرسال سلسلة خطوات T-IR الكاملة إلى نظام المهام في الخلفية (Job Queue) لتنفيذها على كامل البيانات.
  5. عرض شريط تقدم وحالة المهمة للمستخدم.
