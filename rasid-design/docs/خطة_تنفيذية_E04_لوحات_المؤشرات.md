# الخطة التنفيذية الشاملة — محرك لوحات المؤشرات (E04)

## ملخص تنفيذي
تهدف هذه الوثيقة إلى توفير خطة تنفيذية فنية شاملة ومكتفية ذاتياً لإنشاء **محرك لوحات المؤشرات (E04)**. هذا المحرك مصمم لتمكين المستخدمين من بناء لوحات مؤشرات وتحكم على مستوى المؤسسات (Enterprise-grade) من خلال واجهة استخدام بسيطة وموحدة. يدعم المحرك آلاف العناصر المرئية (Widgets) ومؤشرات الأداء الرئيسية (KPIs)، مع إمكانيات ربط حية بمختلف مصادر البيانات، وتنفيذ عمليات تحويل متقدمة. يتيح المحرك للمستخدمين إنشاء اللوحات إما تلقائياً عبر وصف نصي، أو يدوياً باستخدام السحب والإفلات، أو حتى استنساخ لوحات موجودة من صور أو ملفات PDF. ترتكز الخطة على تحقيق كافة المتطلبات الواردة في وثيقة المواصفات، مع ضمان الأداء العالي، والأمان، وقابلية التوسع، وتوفير مخرجات دقيقة وقابلة للتدقيق.

## ثالثاً: الخطة التنفيذية المرحلية

### المرحلة 1: تأسيس البنية التحتية والتمثيل الوسيط (D-IR)

#### المهمة 1.1: إعداد هيكل المشروع وتحديد معايير النجاح
- **المتطلبات المشمولة**:
  - `E04-0001`: **وثيقة تنفيذية فنية موجهة للمنفّذ مباشرة** — صياغة أمرية إلزامية (MUST/SHALL/MUST NOT)
  - `E04-0002`: **هدف المحرك**: إنشاء لوحات مؤشرات/تحكم عالمية (Enterprise-grade) بواجهة بسيطة (Canvas واحد) مع:
  - `E04-0008`: لا كود وهمي/لا ديمو/لا ادعاء بدون دليل وتنفيذ فعلي.
  - `E04-0009`: **قاعدة**: أي سلوك غير منصوص عليه هنا = ممنوع.
  - `E04-0010`: **لا أسئلة**: المحرك لا يسأل المستخدم أثناء البناء/التوليد. أي اختيار يُدار عبر Defaults أو Controls.
  - `E04-0011`: **لا ادّعاء**: لا “تم” إلا بعد Artifact/Link حي + اختبارات + Evidence.
  - `E04-0012`: المحرك يُعتبر ناجحًا فقط إذا حقق **كل** الآتي:
  - `E04-0020`: **No-Cheating**: لا stubs/demos/outputs ثابتة تُعرض كحقيقة، ولا “done” بدون Evidence Pack.
- **الحالة الحالية**: غير منفّذ
- **خطوات التنفيذ**:
  1. إنشاء مستودع الكود المصدري للمشروع.
  2. إعداد بيئات التطوير والاختبار والإنتاج.
  3. كتابة وثيقة المبادئ التوجيهية للمشروع التي تفرض الالتزام بالمتطلبات الإلزامية وعدم استخدام أي كود وهمي.
  4. إعداد نظام تتبع المهام لربط كل مهمة بالمتطلبات الخاصة بها.

#### المهمة 1.2: تصميم وتنفيذ التمثيل الوسيط للوحة المؤشرات (D-IR)
- **المتطلبات المشمولة**:
  - `E04-0050`: **Dashboard IR (D-IR)**: تمثيل داخلي موحد (layout + widgets + bindings + interactions)
  - `E04-0067`: المنفّذ MUST يبني D-IR كما هو. أي نقص = رفض.
  - `E04-0068`: `dashboard_ir` MUST include: version
  - `E04-0069`: dashboard_id
  - `E04-0070`: theme_tokens
  - `E04-0071`: dataset_bindings[]
  - `E04-0072`: semantic_model_ref
  - `E04-0073`: global_filters[]
  - `E04-0074`: parameters[]
  - `E04-0075`: interactions[]
  - `E04-0076`: fingerprints: layout_hash, binding_hash, interaction_hash
  - `E04-0077`: page_id, index, name
  - `E04-0078`: grid_spec (columns/rows/gutters/margins)
  - `E04-0079`: widgets[] ordered by z-index
  - `E04-0080`: navigation (optional)
  - `E04-0081`: kind: kpi \| chart \| table \| pivot \| text \| image \| icon \| slicer \| filter_panel \| map \| gauge \| bullet \| sparkline \| heatmap \| treemap \| waterfall \| funnel \| timeline \| boxplot \| scatter \| combo \| cards_grid \| narrative \| embed \| video
  - `E04-0082`: bbox (x,y,w,h) in grid units, z_index
  - `E04-0083`: style_ref (tokenized)
  - `E04-0084`: data_binding_ref (optional)
  - `E04-0085`: interaction_bindings (optional)
- **الحالة الحالية**: غير منفّذ
- **خطوات التنفيذ**:
  1. تعريف هياكل البيانات (Data Structures) الخاصة بـ D-IR في لغة البرمجة المختارة (e.g., TypeScript/Python).
  2. تطوير وحدات (modules) لإنشاء والتحقق من صحة (validation) هياكل D-IR.
  3. كتابة اختبارات وحدوية (Unit Tests) لضمان تطابق D-IR مع المواصفات بشكل كامل.

#### المهمة 1.3: بناء الوحدات الأساسية للمحرك
- **المتطلبات المشمولة**:
  - `E04-0051`: **Layout/Grid Engine** (deterministic)
  - `E04-0060`: **Versioning & Audit** (immutable logs)
  - `E04-0064`: كل خطوة = Action
  - `E04-0065`: كل Tool = Schema input/output
  - `E04-0066`: كل نتيجة = ArtifactRef/LinkRef + fingerprints
  - `E04-0235`: كل Tool MUST يستخدم شكل موحد: request_id/tool_id/context/inputs/params.
  - `E04-0236`: Draft 2020-12.
  - `E04-0237`: **ملاحظة**: هذه “مجموعة حد أدنى تشغيلية” تغلق المحرك وتمنع الاجتهاد.
- **الحالة الحالية**: غير منفّذ
- **خطوات التنفيذ**:
  1. تطوير محرك التخطيط (Layout Engine) الذي يقوم بترتيب العناصر على الشبكة (Grid) بشكل حتمي (deterministic).
  2. بناء نظام تسجيل الإجراءات (Action Runtime) الذي يتعامل مع كل عملية كتصرف (Action) له مدخلات ومخرجات محددة.
  3. تصميم وتنفيذ نظام إدارة الإصدارات (Versioning) الذي يسجل كل تغيير على D-IR كسجل غير قابل للتغيير.

### المرحلة 2: استيعاب البيانات ومعالجتها

#### المهمة 2.1: تطوير موصلات مصادر البيانات
- **المتطلبات المشمولة**:
  - `E04-0034`: Files: XLSX/CSV/TXT
  - `E04-0035`: Documents: PDF (tables), Images (table/dashboard screenshots)
  - `E04-0036`: Connectors: Drive/OneDrive/SharePoint/S3 + DBs (Postgres/MySQL/SQL Server/BigQuery/Snowflake…) (إذا المنصة توفرها)
  - `E04-0037`: Semantic Model / Lakehouse tables
  - `E04-0038`: أي مصدر MUST يتحول إلى: Columnar storage (Arrow/Parquet), Catalog stats, Lineage metadata, Semantic tags
- **الحالة الحالية**: غير منفّذ
- **خطوات التنفيذ**:
  1. تطوير وحدات قارئة للملفات (XLSX, CSV, TXT).
  2. بناء وحدة لاستخلاص الجداول من ملفات PDF والصور.
  3. تطوير موصلات لقواعد البيانات الشائعة ومنصات التخزين السحابي.
  4. إنشاء عملية موحدة (Universal Intake Rule) لتحويل جميع البيانات المستوردة إلى صيغة عمودية (Apache Arrow/Parquet) وتوليد البيانات الوصفية اللازمة.

#### المهمة 2.2: بناء محركات تحويل وقياس البيانات
- **المتطلبات المشمولة**:
  - `E04-0004`: ربط حي بأي مصدر بيانات (ملفات/قاعدة بيانات/موصلات) + عمليات Transform (T-IR) + Measures (M-IR)
  - `E04-0052`: **Data Binding Engine** (columns/measures/joins/filters)
  - `E04-0053`: **Transform Engine (T-IR)** (PowerQuery-like)
  - `E04-0054`: **Measures Engine (M-IR)** (DAX-like; أو equivalent)
  - `E04-0055`: **Query Engine** (columnar + caching + materialized views)
  - `E04-123`: T-IR MUST support: Source (table/view), Promote Headers, Remove/Choose Columns, Filter Rows, ...
  - `E04-124`: ... Add Custom Column, Conditional Column, Group By, Merge, Append, ...
  - `E04-125`: ... Pivot/Unpivot, Transpose, ...
  - `E04-126`: M-IR MUST support: SUM, AVERAGE, MIN, MAX, COUNT, DISTINCTCOUNT, ...
  - `E04-127`: ... CALCULATE, FILTER, ALL, EARLIER, ...
  - `E04-128`: ... Time intelligence (YTD, MTD, QTD, SAMEPERIODLASTYEAR)
- **الحالة الحالية**: غير منفّذ
- **خطوات التنفيذ**:
  1. تصميم وتنفيذ التمثيل الوسيط للتحويلات (T-IR) والقياسات (M-IR).
  2. تطوير محرك التحويلات (Transform Engine) لتنفيذ العمليات المحددة (مثل Group By, Merge, Pivot).
  3. تطوير محرك القياسات (Measures Engine) لدعم الدوال الحسابية والإحصائية ودوال ذكاء الوقت.
  4. بناء محرك الاستعلام (Query Engine) الذي يعمل على البيانات العمودية ويدعم التخزين المؤقت (Caching).

### المرحلة 3: بناء الواجهات والعناصر المرئية

#### المهمة 3.1: تطوير واجهة المستخدم الرئيسية
- **المتطلبات المشمولة**:
  - `E04-0013`: **Canvas واحد**: لوحة واحدة + Panel واحد قابل للإخفاء (لا صفحات إعدادات كثيرة).
  - `E04-0021`: **Dashboard Canvas** (المساحة الرئيسية): grid + widgets + guides.
  - `E04-0022`: **Side Panel واحد** (قابل للإخفاء) يضم: Library, Widget Picker, Data Picker, Filters, Styles, Pages, History, Share, Export.
  - `E04-0023`: **Preview Reader** (Overlay / نافذة): تشغيل اللوحة كقارئ حي مع تفاعل.
  - `E04-0024`: MUST عدم عرض خيارات كثيرة دفعة واحدة.
  - `E04-0025`: MUST عرض 5–9 Controls حسب السياق.
  - `E04-0026`: MUST وجود **Search Controls** لإظهار أي خيار بالبحث.
- **الحالة الحالية**: غير منفّذ
- **خطوات التنفيذ**:
  1. بناء المكونات الرئيسية لواجهة المستخدم (Canvas, Side Panel) باستخدام إطار عمل مناسب (e.g., React).
  2. تطوير آلية التحكم التدريجي (Progressive Controls) التي تعرض الخيارات بناءً على سياق المستخدم.
  3. إضافة شريط البحث (Search Controls) في جميع القوائم لتسهيل الوصول.

#### المهمة 3.2: تطوير كتالوج العناصر المرئية (Widgets)

  - `E04-0091`: Table widget MUST include: columns[], sort/filter, conditional formatting rules

  - `E04-103`: Map widget MUST include: layers, bubble/choropleth, tooltips, legend

  - `E04-139`: MUST support variants (e.g., Bar chart with rounded corners).
  - `E04-140`: Parametric generators MUST create widgets from spec (e.g., “KPI grid 3x2”).
- **الحالة الحالية**: غير منفّذ
- **خطوات التنفيذ**:

  3. إنشاء مولدات بارامترية (Parametric Generators) لإنشاء تكوينات معقدة من العناصر برمجيًا.

#### المهمة 3.3: بناء محرك التصيير (Rendering Engine)

  - `E04-0029`: UI MUST لا يتجمد.

- **الحالة الحالية**: غير منفّذ
- **خطوات التنفيذ**:

  2. تطبيق آلية المعاينة الفورية (Live Preview) للتغييرات على عينة من البيانات.
  3. تنفيذ المهام غير المتزامنة (Async Jobs) لتطبيق التغييرات على مجموعة البيانات الكاملة دون تجميد واجهة المستخدم.

  - `E04-118`: Cross-filtering MUST be supported between compatible widgets.
  - `E04-119`: Drill-down/up MUST be configurable on hierarchies.

- **الحالة الحالية**: غير منفّذ

  4. تطوير نظام الإشارات المرجعية (Bookmarks) لحفظ حالات العرض المختلفة.

#### المهمة 4.2: تطوير نظام المشاركة والصلاحيات
