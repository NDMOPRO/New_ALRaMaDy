## المراحل الإضافية: المتطلبات المكملة

### المرحلة 50: تأسيس واجهة المستخدم الموحدة (Single Canvas UI)
#### المهمة 50.1: فرض قيود صارمة على الواجهة
- **المتطلبات المشمولة**:
  - `E07-0002`: **الهدف**: واجهة واحدة فقط (Canvas) تعمل كـChat + مساحة عمل ديناميكية تُظهر *كل النتائج/التحرير/المعاينة/التصدير* داخل نفس الشاشة بدون فتح صفحات.
  - `E07-0003`: **قاعدة صارمة**: أي سلوك غير منصوص عليه هنا = **ممنوع**. أي انتقال لصفحات/تشتيت = **مرفوض**.
  - `E07-0004`: **Canvas واحد فقط**: لا “صفحات أدوات”. لا “اذهب إلى صفحة الإكسل/اللوحات/التقارير”. كل شيء يظهر داخل Canvas.
- **الحالة الحالية**: غير منفّذ
- **خطوات التنفيذ**:
  1. تصميم المكون الرئيسي للتطبيق كـ "Canvas" وحيد لا يتغير.
  2. منع أي استخدام لمكتبات التوجيه (Routing) التي تؤدي إلى تغيير عنوان URL أو تحميل صفحة جديدة.
  3. تضمين آلية لمراجعة الكود تمنع أي إضافة لسلوكيات غير معتمدة في المتطلبات.

#### المهمة 50.2: تفعيل نمط التحكم عبر المحادثة (Chat-Led Control)
- **المتطلبات المشمولة**:
  - `E07-0005`: **Chat هو القائد**: كل عملية تبدأ من رسالة/أمر داخل الشات أو Drag&Drop إلى الشات.
  - `E07-0006`: **Dynamic Context**: الخيارات لا تظهر إلا حسب سياق (نوع الملف + ما حدده المستخدم + المرحلة الحالية).
  - `E07-0007`: **Progressive Disclosure**: ممنوع إغراق المستخدم بخيارات مرة واحدة.
- **الحالة الحالية**: غير منفّذ
- **خطوات التنفيذ**:
  1. ربط جميع الإجراءات الأولية (بدء العمليات) بحقل إدخال المحادثة (Composer) أو بمنطقة السحب والإفلات (Drag & Drop).
  2. تطوير محرك سياق (Context Engine) يحلل الحالة الحالية (الملفات المرفوعة، الأوامر السابقة) لتحديد الإجراءات المتاحة.
  3. تصميم واجهة الإجراءات (ContextActionsCard) بحيث لا تعرض أكثر من 7 خيارات وتستخدم زر "More" لعرض خيارات متقدمة عند الحاجة.

### المرحلة 60: تطبيق آلة الحالة المحدودة (Finite State Machine)
#### المهمة 60.1: تصميم وتنفيذ آلة الحالة
- **المتطلبات المشمولة**:
  - `E07-0207`: **الهدف:** تحويل مفهوم “Canvas واحد + Chat + Sidebar اختياري + Focus Stage + Preview Reader + Tours + Jobs” إلى **Finite State Machines** دقيقة (States + Events + Guards + Actions + Invariants) جاهزة للتطبيق في الواجهة.
  - `E07-0208`: **تنفيذ إلزامي:**
  - `E07-0209`: - الواجهة MUST تُنفّذ هذا الـState Chart حرفيًا باستخدام **XState** (مفضل) أو FSM مكافئ.
  - `E07-0210`: - إذا لم تستخدم XState، MUST تُقدّم اختبارات “State Equivalence” تُثبت أن السلوك مطابق 1:1 لكل Transition.
  - `E07-0211`: هذه نقاط مُلاحظة مباشرة من ملفات الواجهة داخل ZIP (Home.tsx / NavDesigns.tsx / DesignContext / ThemeContext / ParticleBackground):
  - `E07-0212`: حالة Theme: `light \| dark` + toggle.
  - `E07-0213`: حالة Panel/Overlay: `isPanelOpen` (لوحة تصميم/تحكم) + toggle.
  - `E07-0214`: حالة Navigation: `activePage` + `expandedItems[]` + `sidebarCollapsed`.
  - `E07-0215`: حالة Chat: `messages[]` + `isTyping`
  - `E07-0217`: **ملاحظة:** هذه الـStates تُعاد صياغتها هنا لتصبح “Production State Machine” للـCanvas الواحد.
  - `E07-0218`: **Single Canvas:** لا يوجد Route-level navigation لصفحات أدوات. أي “صفحة” تُعرض كـView داخل نفس الـCanvas.
  - `E07-0219`: **One Focus Stage:** لا يمكن فتح أكثر من Focus Stage واحد في نفس اللحظة.
  - `E07-0220`: **One Modal Overlay:** لا يمكن أن تكون أكثر من Overlay “Blocking Modal” مفتوحة في نفس اللحظة (مانع تفاعل).
  - `E07-0221`: **Sidebar لا يحجب Canvas:** Sidebar لا يُغطي كامل الشاشة إلا في Mobile Overlay Mode.
  - `E07-0222`: **No Dead-End Rule:** عند أي حالة (حتى فشل) MUST يظهر Next Valid Action (Retry / Repair / Export logs / Open evidence).
  - `E07-0223`: **Truthful UI:** ممنوع إظهار “Completed/Done” قبل استلام `evidence_id` و`artifact_ids` من الـAction Runtime.
  - `E07-0224`: Chat stream MUST يكون Virtualized.
  - `E07-0225`: Preview/Render MUST تكون Async jobs ولا تجمّد الـUI.
  - `E07-0226`: Animations MUST احترام Reduce Motion.
- **الحالة الحالية**: غير منفّذ
- **خطوات التنفيذ**:
  1. تصميم آلة الحالة المحدودة (FSM) باستخدام XState أو مكتبة مكافئة.
  2. تعريف جميع الحالات (States)، والأحداث (Events)، والشروط (Guards)، والإجراءات (Actions).
  3. التأكد من أن تنفيذ الواجهة يتبع مخطط الحالة بدقة.
  4. كتابة اختبارات للتأكد من تطابق سلوك الواجهة مع مخطط الحالة.
