# الخطة التنفيذية الشاملة — محرك التعريب والتحويل والتفريغ (LCT)

## ملخص تنفيذي
هذا المستند يقدم الخطة التنفيذية الفنية الشاملة لمحرك التعريب والتحويل والتفريغ (LCT Engine - E06). المحرك مصمم ليكون حلاً مركزياً لمعالجة وتحويل طيف واسع من أنواع الملفات والبيانات. الهدف الأساسي هو استقبال أي نوع من المدخلات (مثل PDF، صور، ملفات صوتية، فيديو، مستندات Office) وتحويلها إلى أي صيغة مطلوبة (مثل PPTX, DOCX, XLSX, JSON, SRT) مع ضمان الدقة المطلقة والجودة الاحترافية. تعمل الخطة وفق ثلاثة محاور رئيسية: التحويل البصري المطابق 100% (CONVERT_STRICT_1TO1_100)، والتعريب العالمي الاحترافي (LOCALIZE_PRO_100)، والتفريغ النصي الدقيق 100% (TRANSCRIBE_STRICT_100). لضمان عدم وجود أي هامش للخطأ، تعتمد الخطة على بوابات تحقق آلية صارمة، ونظام تدقيق بشري داخلي (VerifierOps) لمعالجة أي حالة عدم يقين دون إزعاج المستخدم، وتوليد حزمة أدلة (Evidence Pack) لكل عملية لضمان الشفافية والمساءلة.

## ثالثاً: الخطة التنفيذية المرحلية

### المرحلة 1: الأساسيات والبنية التحتية

#### المهمة 1.1: تعريف هوية المحرك وقواعده الأساسية
- **المتطلبات المشمولة**:
  - `E06-0001`: **وثيقة تنفيذية فنية موجهة للمنفّذ مباشرة** — صياغة أمرية إلزامية (MUST/SHALL/MUST NOT)
  - `E06-0002`: **اسم المحرك**: LCT = Localization + Conversion + Transcription Engine
  - `E06-0003`: **الهدف**: “أعطني أي شيء” (PDF/صورة/فيديو/صوت/Word/Excel/PowerPoint/نص…) → “أعطني أي شيء” (PPTX/DOCX/XLSX/Dashboard/PDF/HTML/PNG/SRT/VTT/JSON…)
  - `E06-0008`: **لا كود وهمي/لا ديمو/لا ادعاء**: لا “Done” إلا بعد artifacts حقيقية + gates + evidence.
  - `E06-0009`: **قاعدة صارمة**: أي سلوك غير منصوص عليه هنا = **ممنوع**.
  - `E06-0010`: **لا أسئلة للمستخدم**: المحرك لا يطرح أسئلة أثناء التنفيذ. أي نقص/غموض يُحل بسياسات افتراضية أو يُعالج داخليًا (VerifierOps).
  - `E06-0104`: **MUST enforce**: gates block merge/release, no stubs/mocks/TODO in runtime, deterministic builds where claims depend on parity.
- **الحالة الحالية**: غير منفّذ
- **خطوات التنفيذ**:
  1. إعداد مستودع الكود المصدري للمحرك.
  2. كتابة وثيقة `README.md` تعرف بالمحرك وأهدافه وقواعده الأساسية.
  3. إعداد بيئة التطوير وإطار العمل للاختبارات الآلية (Automated Testing Framework) لفرض القواعد (Gates) عند كل عملية دمج (Merge).
  4. تعريف السياسات الافتراضية الأولية التي سيستخدمها المحرك في حالة وجود غموض في المدخلات.

#### المهمة 1.2: تصميم وتنفيذ الواجهة الموحدة (UI/UX)
- **المتطلبات المشمولة**:
  - `E06-0026`: Canvas واحد: drop أي ملف/رابط/تسجيل
  - `E06-0027`: Panel واحد قابل للإخفاء: Inputs (Library/Connectors), Task (Convert / Localize / Transcribe / Extract Structured), Outputs, Policies (STRICT claims toggles), Preview Reader (overlay), History/Versions, Share/Permissions/Classification
  - `E06-0028`: لا تُعرض خيارات كثيرة دفعة واحدة.
  - `E06-0029`: أي خيار إضافي يظهر **حسب السياق** + عبر Search Controls.
- **الحالة الحالية**: غير منفّذ
- **خطوات التنفيذ**:
  1. تصميم نماذج أولية (Mockups) للواجهة الرئيسية التي تتضمن الـ Canvas والـ Panel.
  2. تطوير الواجهة الأمامية (Frontend) باستخدام إطار عمل حديث (مثل React أو Vue).
  3. تنفيذ منطق التحكمات التدريجية (Progressive Controls) بحيث لا تظهر الخيارات إلا عند الحاجة إليها.
  4. ربط الواجهة بمكتبة الملفات ومصادر الإدخال الأولية (Connectors).

#### المهمة 1.3: بناء التمثيل الموحد للبيانات والتصميم (CDR)
- **المتطلبات المشمولة**:
  - `E06-0030`: كل التحويلات MUST تمر عبر **CDR** (Canonical Design/Data Representation) الخاص بالمنصة: CDR-Design (layout + typography + vectors + images), CDR-Data (tables + semantic bindings)
  - `E06-0031`: ممنوع مسارات خاصة لكل صيغة bypass.
- **الحالة الحالية**: غير منفّذ
- **خطوات التنفيذ**:
  1. تصميم المخططات (Schemas) الخاصة بـ CDR-Design و CDR-Data لتمثيل جميع العناصر البصرية والبيانية بشكل موحد.
  2. تطوير المكتبات البرمجية (Libraries) اللازمة لقراءة أنواع الملفات المختلفة (PDF, DOCX, etc.) وتحويلها إلى صيغة CDR.
  3. تطوير المكتبات البرمجية اللازمة لتصدير البيانات من صيغة CDR إلى أنواع الملفات المختلفة (PPTX, HTML, etc.).
  4. بناء مسار المعالجة (Pipeline) الإلزامي الذي يضمن مرور جميع عمليات التحويل عبر CDR.

### المرحلة 2: تطوير قدرات التحويل الدقيق (Conversion)

#### المهمة 2.1: تنفيذ شارة التحويل المطابق بصرياً (CONVERT_STRICT_1TO1_100)
- **المتطلبات المشمولة**:
  - `E06-0012`: المحرك يدعم 3 “Claims” منفصلة. يمنع خلطها:
  - `E06-0013`: **CONVERT_STRICT_1TO1_100**
  - `E06-0014`: التحويل/الاستنساخ البصري: `PixelDiff == 0` لكل صفحة/شريحة داخل Farm حتمية.
  - `E06-0015`: الناتج MUST يكون Editable Core (نص/جداول/مخططات/عناصر) — ممنوع “كل شيء صورة”.
  - `E06-0040`: إذا تم اختيار CONVERT_STRICT_1TO1_100: MUST, MUST NOT
  - `E06-0041`: صور الطبيعة (photos/logos) مسموحة كأصول فقط، وتبقى قابلة للتحرير (crop/replace/mask).
- **الحالة الحالية**: غير منفّذ
- **خطوات التنفيذ**:
  1. إعداد بيئة حتمية (Deterministic Environment) أو ما يعرف بـ "Farm" لضمان أن تكون عمليات تصيير (Rendering) المدخلات والمخرجات متطابقة تماماً.
  2. تطوير بوابة التحقق (Gate) التي تقوم بتصيير الملف المصدر والملف الناتج كصور ومقارنتها باستخدام `PixelDiff`.
  3. التأكد من أن بوابة التحقق تمنع إتمام العملية إذا كان `PixelDiff` لا يساوي صفراً.
  4. تطوير آلية للتحقق من أن المخرجات قابلة للتحرير (Editable) وليست مجرد صور مدمجة.

#### المهمة 2.2: بناء مسارات التحويل الأساسية
- **المتطلبات المشمولة**:
  - `E06-0032`: المحرك MUST يدعم (على الأقل):
  - `E06-0033`: PDF ↔ PPTX (Editable)
  - `E06-0034`: PDF ↔ DOCX (Editable)
  - `E06-0035`: Image ↔ XLSX (table-strict)
  - `E06-0036`: Image ↔ PPTX (strict slide build)
  - `E06-0037`: XLSX/CSV → Dashboard
  - `E06-0038`: Dashboard → PPTX/DOCX/PDF/HTML
- **الحالة الحالية**: غير منفّذ
- **خطوات التنفيذ**:
  1. تنفيذ مسار التحويل بين PDF و PPTX القابل للتحرير، مع التركيز على الحفاظ على التنسيق والعناصر.
  2. تنفيذ مسار التحويل بين PDF و DOCX القابل للتحرير.
  3. تطوير قدرات استخراج الجداول من الصور وتحويلها إلى ملفات XLSX.
  4. تطوير قدرات إعادة بناء الشرائح (PPTX) من الصور.
  5. بناء مسارات التحويل من وإلى لوحات المعلومات (Dashboards).
  6. ربط جميع المسارات ببوابة التحقق `CONVERT_STRICT_1TO1_100`.

### المرحلة 3: تطوير قدرات التعريب الاحترافي (Localization)

#### المهمة 3.1: تنفيذ شارة التعريب الاحترافي (LOCALIZE_PRO_100)
- **المتطلبات المشمولة**:
  - `E06-0016`: **LOCALIZE_PRO_100** (تعريب عالمي احترافي)
  - `E06-0017`: ترجمة مصطلحية سياقية + التزام قالب/أسلوب + Arabic ELITE typesetting.
  - `E06-0018`: **LQA==0** (لا أخطاء مصطلح/معنى/نبرة/اتساق/أسماء/أرقام/وحدات).
  - `E06-0019`: Layout MUST يبقى محافظًا على التصميم (layout lock) مع إصلاحات هندسية لا تُفسد الترتيب.
  - `E06-0042`: التعريب هنا يعني: ترجمة النصوص + المصطلحات + الأرقام/الوحدات + السياق الإداري, إعادة بناء layout RTL ELITE, تكييف الجداول/المخططات/المحاور/التسميات, احترام القالب/النبرة (حكومي/تجاري/تنفيذي/تقني), حفظ التصميم قدر الإمكان دون تشويه (layout lock).
- **الحالة الحالية**: غير منفّذ
- **خطوات التنفيذ**:
  1. تطوير بوابة ضمان الجودة اللغوية (LQA Gate) التي تتحقق من عدم وجود أخطاء في الترجمة.
  2. تطوير بوابة التحقق من الحفاظ على التصميم (Layout Lock Gate).
  3. دمج محرك تنضيد متقدم للغة العربية (Arabic ELITE Typesetting Engine) يدعم bidi/shaping.

#### المهمة 3.2: بناء أنظمة المصطلحات والذاكرة والأسلوب
- **المتطلبات المشمولة**:
  - `E06-0048`: MUST وجود: **Termbase** (مصطلحات عربية/إنجليزية) مع domains (مالي/حكومي/تقني…), **Translation Memory** per tenant, **Style Guide** per tenant (tone, honorifics, formal phrases, forbidden phrases)
- **الحالة الحالية**: غير منفّذ
- **خطوات التنفيذ**:
  1. تصميم وتنفيذ قاعدة بيانات للمصطلحات (Termbase) قابلة للتصنيف حسب المجال (Domain).
  2. بناء نظام ذاكرة الترجمة (Translation Memory) على مستوى كل مستأجر (Tenant).
  3. تطوير نظام أدلة الأسلوب (Style Guides) لتحديد النبرة والعبارات المسموحة والمحظورة.
  4. ربط هذه الأنظمة بعملية التعريب لضمان الاتساق والجودة.

#### المهمة 3.3: تنفيذ محرك إصلاح التصميم التلقائي (Layout Repair)
- **المتطلبات المشمولة**:
  - `E06-0049`: MUST استخدام Arabic shaping/bidi pinned engine
  - `E06-0050`: MUST تحويل: charts axes RTL, legends RTL, table direction RTL, numbering style per policy (٠١٢٣ أو 0123)
  - `E06-0051`: التعريب MUST لا “يكسر” التصميم: ممنوع overflow/clipping, ممنوع تداخل عناصر
  - `E06-0052`: MUST تنفيذ Layout Repair loop: text box resize rules deterministic, font size stepdown deterministic, reflow داخل حدود bbox فقط, grid snap
- **الحالة الحالية**: غير منفّذ
- **خطوات التنفيذ**:
  1. تنفيذ خوارزميات حتمية لإصلاح التصميم تلقائياً عند تحويل الواجهة لتدعم الكتابة من اليمين لليسار (RTL).
  2. تطوير قواعد لتغيير حجم مربعات النص، وتصغير حجم الخط بشكل تدريجي، وإعادة تدفق النص ضمن حدوده.
  3. تطبيق القواعد على الجداول والمخططات والمحاور لضمان عرضها بشكل صحيح في نسخة RTL.

#### المهمة 3.4: بناء بوابات التحقق الخاصة بالتعريب
- **المتطلبات المشمولة**:
  - `E06-0054`: Terminology compliance = 100%
  - `E06-0055`: LQA = 0 أخطاء (مصطلح/معنى/اتساق/نبرة/أسماء/أرقام/وحدات)
  - `E06-0056`: Layout QA pass (no overlap/no clip)
  - `E06-0057`: Arabic ELITE pass
  - `E06-0058`: Evidence Pack stored
  - `E06-0059`: ثم فقط: “Localized Successfully”.
- **الحالة الحالية**: غير منفّذ
- **خطوات التنفيذ**:
  1. تطوير بوابة تحقق من الالتزام بالمصطلحات (Terminology Compliance Gate).
  2. تطوير بوابة جودة التصميم (Layout QA Gate) التي تفحص بحثاً عن أي تداخل أو تجاوز للحدود.
  3. تطوير بوابة للتحقق من تطبيق قواعد التنضيد العربي المتقدم (Arabic ELITE Gate).
  4. تجميع نتائج جميع البوابات في حزمة الأدلة (Evidence Pack) كشرط لإعلان نجاح عملية التعريب.

### المرحلة 4: تطوير قدرات التفريغ النصي الدقيق (Transcription)

#### المهمة 4.1: تنفيذ شارة التفريغ الدقيق (TRANSCRIBE_STRICT_100)
- **المتطلبات المشمولة**:
  - `E06-0020`: **TRANSCRIBE_STRICT_100** (تفريغ/نسخ/استخراج نص 100%)
  - `E06-0021`: transcript MUST يكون “Exact” وفق بوابات تحقق متعددة، و**لا يُسلّم** قبل الوصول للـExact.
  - `E06-0023`: لا يُسمح بإرجاع “قريب” أو “confidence 0.97” كناتج نهائي. إما Exact أو لا تسليم.
  - `E06-0024`: **ممنوع**: أي “99.9999” أو “قريب جدًا”.
  - `E06-0079`: ممنوع تسليم “best effort”. لا يوجد “تقريب”.
- **الحالة الحالية**: غير منفّذ
- **خطوات التنفيذ**:
  1. تعريف معنى "Exact" للتفريغ النصي وتحديد معايير النجاح الصارمة.
  2. تصميم بنية النظام بحيث لا يتم تسليم أي نتيجة للمستخدم إلا بعد اجتياز جميع بوابات التحقق والوصول إلى حالة "Exact".
  3. إعداد آلية التوجيه التلقائي إلى فريق التحقق البشري (VerifierOps) عند فشل أي بوابة.

#### المهمة 4.2: دعم وسائط الإدخال والمخرجات المتعددة
- **المتطلبات المشمولة**:
  - `E06-0039`: Video/Audio → Transcript (DOCX/JSON/SRT/VTT) + optional “slides/report” outputs
  - `E06-0060`: المحرك MUST يدعم التفريغ من: صور (OCR text), PDF (text + scanned OCR), Audio (ASR), Video (ASR + on-screen text OCR + embedded captions), Camera capture (صورة/فيديو) كـinput asset
  - `E06-0066`: DOCX transcript
  - `E06-0067`: JSON transcript (segments, speakers, timestamps, confidence fields for audit only)
  - `E06-0068`: SRT/VTT subtitles
  - `E06-0069`: “Screen text extraction” (on-screen overlays)
  - `E06-0070`: “Comments extraction”: إذا الفيديو يحتوي track captions/metadata/comments داخل الملف أو ضمن منصة موصلات: MUST استخراجها. إذا غير متوفر مصدر: MUST لا يدّعي وجودها
- **الحالة الحالية**: غير منفّذ
- **خطوات التنفيذ**:
  1. دمج محركات التعرف الضوئي على الحروف (OCR) لاستخراج النصوص من الصور وملفات PDF الممسوحة ضوئياً.
  2. دمج محركات التعرف على الكلام (ASR) لتفريغ الملفات الصوتية وملفات الفيديو.
  3. تطوير القدرة على استخراج النصوص الظاهرة على الشاشة في مقاطع الفيديو.
  4. بناء وحدات التصدير (Exporters) لإنشاء ملفات DOCX, JSON, SRT/VTT من النص المفرّغ.

#### المهمة 4.3: بناء بوابات التحقق الخاصة بالتفريغ (Transcription Gates)
- **المتطلبات المشمولة**:
  - `E06-0071`: النجاح 100% لا يتم عبر “confidence”، بل عبر gates:
  - `E06-0072`: **Multi-Engine Consensus**: run ≥2 ASR engines pinned, اختلافات تُحدد تلقائيًا
  - `E06-0073`: **Forced Alignment Gate**: transcript MUST align perfectly to audio (token→time)
  - `E06-0074`: **Round-Trip Validation Gate**: validate numbers, names, acronyms via constrained decoding + glossary
  - `E06-0075`: **Video OCR Gate** (للفيديو/الشاشة): extract on-screen text, compare with spoken text for contradictions (if relevant)
  - `E06-0076`: **Exactness Gate**: إذا بقيت أي “uncertain spans” => MUST route to VerifierOps
  - `E06-0078`: ثم فقط: TRANSCRIBE_STRICT_100 success.
- **الحالة الحالية**: غير منفّذ
- **خطوات التنفيذ**:
  1. تنفيذ بوابة الإجماع متعدد المحركات (Multi-Engine Consensus) عن طريق تشغيل محركي ASR على الأقل ومقارنة النتائج.
  2. تطوير بوابة المواءمة القسرية (Forced Alignment Gate) للتحقق من تطابق التوقيت الزمني للنص مع الصوت.
  3. بناء بوابة التحقق ذهاباً وإياباً (Round-Trip Validation) للتحقق من صحة الأرقام والأسماء والمختصرات.
  4. تطوير بوابة OCR للفيديو لمقارنة النص المسموع مع النص المرئي.
  5. تنفيذ بوابة الدقة (Exactness Gate) التي تقوم بتوجيه أي مقاطع غير مؤكدة إلى VerifierOps.

### المرحلة 5: أنظمة التحقق والدعم

#### المهمة 5.1: بناء نظام التدقيق البشري الداخلي (VerifierOps)
- **المتطلبات المشمولة**:
  - `E06-0022`: إذا لم تحقق النماذج الآلية Exact، MUST تُرسل المهمة تلقائيًا إلى **VerifierOps** الداخلي (تدقيق بشري داخل المنصة) للوصول إلى Exact **بدون سؤال المستخدم**.
  - `E06-0025`: **Never-Fail للمستخدم** يتحقق داخليًا عبر VerifierOps وليس عبر تخفيض الدقة.
  - `E06-0053`: إذا لم يصل Layout إلى “No-Break QA” آليًا: MUST يذهب إلى VerifierOps (داخلي) لعمل micro-adjustments, بدون سؤال المستخدم
  - `E06-0077`: **VerifierOps Completion Gate**: final transcript MUST be exact (no uncertain markers)
  - `E06-0080`: لتحقيق “100% بدون استثناء” في OCR/ASR/Localization، MUST وجود طبقة تدقيق بشرية داخلية.
  - `E06-0081`: يُستدعى VerifierOps تلقائيًا عند: أي Gate fail في STRICT_100, أي uncertainty span > 0, أي Terminology/LQA violation, أي Layout break غير قابل للإصلاح الآلي
  - `E06-0082`: VerifierOps MUST: يعمل على واجهة “diff-based” (يشاهد فقط المناطق المتنازع عليها), يوقع النتيجة (verifier_id + timestamp), ينتج Evidence داخل pack
  - `E06-0083`: VerifierOps MUST NOT يتواصل مع المستخدم.
  - `E06-0084`: إذا نقصت بيانات مصطلحات، MUST يستخدم termbase الافتراضية + style guide العام.
- **الحالة الحالية**: غير منفّذ
- **خطوات التنفيذ**:
  1. تصميم وتطوير واجهة المستخدم الخاصة بـ VerifierOps، والتي تعرض فقط الاختلافات أو المناطق غير المؤكدة للمدقّق.
  2. بناء نظام الطابور (Queueing System) الذي يستقبل المهام تلقائياً من بوابات التحقق الفاشلة.
  3. تنفيذ آلية التوقيع الرقمي لكل عملية تدقيق.
  4. التأكد من أن النظام لا يسمح بأي تواصل مباشر بين المدقق والمستخدم النهائي.
  5. إنشاء بوابة إتمام التحقق (VerifierOps Completion Gate) التي تضمن أن المخرج النهائي من التدقيق البشري دقيق تماماً.

#### المهمة 5.2: تطوير نظام حزمة الأدلة (Evidence Pack)
- **المتطلبات المشمولة**:
  - `E06-0085`: كل عملية LCT MUST تولد Evidence Pack يحتوي:
  - `E06-0086`: input fingerprints (sha256, mime, metadata)
  - `E06-0087`: pipeline action graph snapshot
  - `E06-0088`: tool versions + farm image id
  - `E06-0089`: for conversion: source render + target render, pixel_hashes + PixelDiff report (must 0 in strict), structural/editability report
  - `E06-0090`: for localization: terminology compliance report, LQA report (must 0 errors), RTL/Arabic shaping report, layout QA report
  - `E06-0091`: for transcription: diarization report, alignment report, unresolved spans list (must empty), verifier ops proof (if used)
  - `E06-0092`: artifacts refs (pptx/docx/xlsx/pdf/html/srt/vtt/json)
  - `E06-0093`: No Evidence => MUST NOT say “Done”
- **الحالة الحالية**: غير منفّذ
- **خطوات التنفيذ**:
  1. تصميم مخطط (Schema) لـ Evidence Pack ليحتوي على جميع المعلومات المطلوبة.
  2. تطوير خدمة مركزية مسؤولة عن تجميع وتخزين حزم الأدلة لكل عملية.
  3. دمج كل مرحلة من مراحل المعالجة (Conversion, Localization, Transcription) مع خدمة حزم الأدلة لتسجيل تقاريرها وإثباتاتها.
  4. التأكد من أن حالة "Done" لا تُعلن لأي عملية إلا بعد إنشاء وتخزين حزمة الأدلة الخاصة بها بنجاح.

### المرحلة 6: التكامل والأمان

  2. تطوير طبقة التنسيق (Orchestration Layer) داخل محرك LCT لاستدعاء المحركات الأخرى حسب الحاجة.
